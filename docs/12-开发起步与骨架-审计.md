## C. 必改项

### C.1 【必改】`EnsureCreated()` → `Migrate()`，确保迁移可追溯

**问题位置**：`SeedData.cs:11` 和 `Program.cs:48-52`

**问题描述**：
`SeedData.Initialize()` 第一行调用 `context.Database.EnsureCreated()`。该方法的行为是：

- 数据库不存在 → 直接用 EF model 创建所有表，**不记录 `__EFMigrationsHistory`**
- 数据库已存在 → 什么都不做

当前工作流能跑通是因为先执行了 `dotnet ef database update`（通过迁移建库），然后 `EnsureCreated()` 发现库已存在就跳过。但此顺序不是一个可靠的工作流——任何人在新机器上直接 `dotnet run`（跳过 `dotnet ef database update`）就会创建出一个没有迁移记录的数据库，后续迁移操作会报错。

**修复方案**：

修改 `SeedData.cs`：移除 `EnsureCreated()`，改为仅在 `Program.cs` 中调用 `Migrate()`。

```csharp
// SeedData.cs — 移除第 11 行 context.Database.EnsureCreated();
// SeedData 只负责种子数据注入，不负责建库
public static void Initialize(AppDbContext context)
{
    // 移除: context.Database.EnsureCreated();
    if (context.Seats.Any())
        return;
    // ... 其余种子逻辑不变
}
```

```csharp
// Program.cs — 将 EnsureCreated 替换为 Migrate
using (var scope = app.Services.CreateScope())
{
    var dbContext = scope.ServiceProvider.GetRequiredService<AppDbContext>();
    dbContext.Database.Migrate();          // ← 确保迁移已应用
    SeedData.Initialize(dbContext);        // ← 然后注入种子数据
}
```

**验收标准**：
1. 删除 LocalDB 数据库（`dotnet ef database drop`）
2. 直接 `dotnet run` 应用启动成功
3. `__EFMigrationsHistory` 表存在且有 `InitialCreate` 记录
4. `sqlcmd` 查询 Seat/Reservation/Admin 三张表种子数据完整

**影响范围**：`SeedData.cs`（1 行删除）、`Program.cs`（1 行替换）

