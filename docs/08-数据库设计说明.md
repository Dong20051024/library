## 2. 命名规范

### 2.1 数据库对象命名

| 对象 | 命名规则 | 示例 |
|------|----------|------|
| 表名 | 英文单数，PascalCase | `Seat`、`Reservation`、`Admin` |
| 主键列 | `Id` | 所有表统一，类型 `int` |
| 外键列 | `[关联表名]Id` | `SeatId` |
| 普通列 | PascalCase | `SeatNumber`、`StudentName`、`StartTime` |
| 索引 | `IX_[表名]_[列名]` | `IX_Reservation_SeatId`、`IX_Reservation_Status` |

### 2.2 C# 实体命名

| 对象 | 规则 | 示例 |
|------|------|------|
| Entity 类 | 单数，与表名一致 | `Seat`、`Reservation`、`Admin` |
| 枚举 | PascalCase | `SeatStatus`、`ReservationStatus` |
| 枚举值 | 中文（与业务术语一致） | `SeatStatus.空闲`、`ReservationStatus.待签到` |
| 导航属性 | 单数或复数 | `Reservation.Seat`（单个）、`Seat.Reservations`（集合） |

> 枚举值使用中文的原因：状态值直接出现在 Razor 视图中（如 `@item.Status.ToString()`），中文可读性远优于英文（Free 不明确、WaitingForSignIn 太长）。EF Core 以 int 存储，不影响数据库。

## 3. 三张数据表详细设计

### 3.1 Seat（座位表）

| 字段 | 类型（C#） | 类型（SQL） | 约束 | 说明 |
|------|-----------|-------------|------|------|
| Id | int | `int` | PK, IDENTITY(1,1) | 主键，自增 |
| Floor | string | `nvarchar(20)` | NOT NULL | 楼层（如："3楼"） |
| Area | string | `nvarchar(50)` | NOT NULL | 区域（如："A区"、"靠窗区"） |
| SeatNumber | string | `nvarchar(20)` | NOT NULL | 座位编号（如："A-01"） |
| Status | SeatStatus (int) | `int` | NOT NULL, DEFAULT 0 | 座位状态：0=空闲, 1=已预约, 2=使用中 |
| Facilities | string | `nvarchar(200)` | NOT NULL, DEFAULT '' | 设施标签，逗号分隔（如："有插座,靠窗,电脑位"） |
| IsActive | bool | `bit` | NOT NULL, DEFAULT 1 | 是否启用（禁用后不可预约） |

**SQL 创建语句：**

```sql
CREATE TABLE [Seat] (
    [Id]          INT           IDENTITY(1,1) NOT NULL PRIMARY KEY,
    [Floor]       NVARCHAR(20)  NOT NULL,
    [Area]        NVARCHAR(50)  NOT NULL,
    [SeatNumber]  NVARCHAR(20)  NOT NULL,
    [Status]      INT           NOT NULL DEFAULT 0,
    [Facilities]  NVARCHAR(200) NOT NULL DEFAULT '',
    [IsActive]    BIT           NOT NULL DEFAULT 1
);
```

**唯一约束：** `(Floor, Area, SeatNumber)` 联合唯一，不允许同一楼层同一区域出现两个相同编号的座位。

```csharp
// Fluent API
entity.HasIndex(s => new { s.Floor, s.Area, s.SeatNumber }).IsUnique();
```

**默认值：** Status=0（空闲）、IsActive=1（启用）

### 3.2 Reservation（预约表）

| 字段 | 类型（C#） | 类型（SQL） | 约束 | 说明 |
|------|-----------|-------------|------|------|
| Id | int | `int` | PK, IDENTITY(1,1) | 主键，自增 |
| SeatId | int | `int` | NOT NULL, FK → Seat(Id) | 外键，关联座位 |
| StudentName | string | `nvarchar(50)` | NOT NULL | 预约学生姓名（体验账号用户名） |
| StartTime | DateTime | `datetime2` | NOT NULL | 预约开始时间 |
| EndTime | DateTime | `datetime2` | NOT NULL | 预约结束时间 |
| Status | ReservationStatus (int) | `int` | NOT NULL, DEFAULT 0 | 预约状态：0=待签到, 1=使用中, 2=已完成, 3=已取消 |
| CreatedAt | DateTime | `datetime2` | NOT NULL, DEFAULT GETDATE() | 创建时间 |

**SQL 创建语句：**

```sql
CREATE TABLE [Reservation] (
    [Id]          INT           IDENTITY(1,1) NOT NULL PRIMARY KEY,
    [SeatId]      INT           NOT NULL,
    [StudentName] NVARCHAR(50)  NOT NULL,
    [StartTime]   DATETIME2     NOT NULL,
    [EndTime]     DATETIME2     NOT NULL,
    [Status]      INT           NOT NULL DEFAULT 0,
    [CreatedAt]   DATETIME2     NOT NULL DEFAULT GETDATE(),

    CONSTRAINT [FK_Reservation_Seat] FOREIGN KEY ([SeatId]) REFERENCES [Seat]([Id])
);
```

**外键行为：** `ON DELETE NO ACTION`（禁止删除有预约记录的座位）

**索引设计：**

```sql
-- 查询加速索引
CREATE INDEX [IX_Reservation_SeatId] ON [Reservation]([SeatId]);
CREATE INDEX [IX_Reservation_StudentName] ON [Reservation]([StudentName]);
CREATE INDEX [IX_Reservation_Status] ON [Reservation]([Status]);

-- 时段冲突查询关键索引（复合索引覆盖查询条件）
CREATE INDEX [IX_Reservation_SeatId_TimeRange] ON [Reservation]([SeatId], [StartTime], [EndTime]);
```

**复合索引说明：** `IX_Reservation_SeatId_TimeRange` 用于时段重叠查询。在 `ReservationService.CreateReservation` 中，需要查询同一座位在目标时段内是否有状态为"待签到"或"使用中"的预约记录，这个索引覆盖了 WHERE 条件中的所有三列。

### 3.3 Admin（管理员表）

| 字段 | 类型（C#） | 类型（SQL） | 约束 | 说明 |
|------|-----------|-------------|------|------|
| Id | int | `int` | PK, IDENTITY(1,1) | 主键，自增 |
| Username | string | `nvarchar(50)` | NOT NULL, UNIQUE | 登录账号名 |
| Password | string | `nvarchar(100)` | NOT NULL | 密码（本期明文存储） |
| DisplayName | string | `nvarchar(50)` | NOT NULL | 显示名称 |

**SQL 创建语句：**

```sql
CREATE TABLE [Admin] (
    [Id]          INT           IDENTITY(1,1) NOT NULL PRIMARY KEY,
    [Username]    NVARCHAR(50)  NOT NULL,
    [Password]    NVARCHAR(100) NOT NULL,
    [DisplayName] NVARCHAR(50)  NOT NULL,

    CONSTRAINT [UQ_Admin_Username] UNIQUE ([Username])
);
```

**唯一约束：** `Username` 唯一

### 4.2 枚举值含义对照

| 枚举 | 值 | 业务含义 | 页面显示 | 颜色规范 |
|------|----|----------|----------|----------|
| SeatStatus.空闲 | 0 | 座位空闲可预约 | "空闲" | 绿色 `#27AE60` |
| SeatStatus.已预约 | 1 | 已被预约但未签到 | "已预约" | 琥珀色 `#E67E22` |

### 4.3 数据库存储方式

枚举在数据库中统一以 `int` 存储，不使用字符串：

| 存储方式 | 选型 | 理由 |
|----------|------|------|
| int 存储 | ✅ 采用 | 存储空间小、查询效率高、排序方便；EF Core 默认行为，无需额外配置 |
| string 存储 | ❌ 不采用 | 存储空间大、拼写错误风险、状态变更需要更新已有数据 |
| SQL Server 自建 enum 类型 | ❌ 不采用 | LocalDB 不支持，且 EF Core 不支持 |

**Fluent API 配置：**

```csharp
// EF Core 默认将 enum 映射为 int，不需要显式配置
// 但为了明确意图，可以写注释说明
entity.Property(e => e.Status)
      .HasConversion<int>()
      .HasDefaultValue(SeatStatus.空闲);
```

## 5. 外键关系

### 5.1 关系总览

```
Seat (1) ──────< (N) Reservation
  │                         │
  │                         │
  PK: Id  ───────────── FK: SeatId
```

### 5.2 外键配置

```csharp
// Reservation 配置中
entity.HasOne(r => r.Seat)
      .WithMany(s => s.Reservations)
      .HasForeignKey(r => r.SeatId)
      .OnDelete(DeleteBehavior.NoAction);  // 禁止级联删除
```

### 5.3 为什么不用级联删除

| 场景 | 行为 |
|------|------|
| 删除座位 → 自动删除预约 | ❌ 禁止。会产生"学生突然发现预约记录消失"的问题 |
| 删除座位 → 拒绝删除 | ✅ 采用。业务层校验 + 外键约束双重保证 |
| 删除管理员 | 🚫 本期不涉及管理员删除操作 |

### 5.4 导航属性完整定义

```csharp
// Seat.cs — 导航属性
public ICollection<Reservation> Reservations { get; set; } = new List<Reservation>();
// 注意：使用 ICollection<T> 而不是 List<T>，EF Core 延迟加载和集合操作更灵活

// Reservation.cs — 导航属性
public Seat Seat { get; set; } = null!;
// 注意：设为 null! 告诉编译器这个属性不会被 null 初始化（EF Core 会填充）
```

## 6. 索引设计

### 6.1 索引清单

| 表 | 索引名 | 列 | 类型 | 用途 |
|----|--------|----|------|------|
| Seat | `PK_Seat` | Id | 聚集索引（主键默认） | 主键查找 |
| Seat | `IX_Seat_Floor_Area_SeatNumber` | Floor, Area, SeatNumber | 唯一非聚集索引 | 防止座位重复 |
| Reservation | `PK_Reservation` | Id | 聚集索引（主键默认） | 主键查找 |
| Reservation | `IX_Reservation_SeatId` | SeatId | 非聚集索引 | 按座位查预约 |
| Reservation | `IX_Reservation_StudentName` | StudentName | 非聚集索引 | "我的预约"查询 |
| Reservation | `IX_Reservation_Status` | Status | 非聚集索引 | 管理端按状态筛选 |
| Reservation | `IX_Reservation_SeatId_TimeRange` | SeatId, StartTime, EndTime | 非聚集索引 | 时段冲突查询 |
| Admin | `PK_Admin` | Id | 聚集索引（主键默认） | 主键查找 |
| Admin | `UQ_Admin_Username` | Username | 唯一约束 | 登录名唯一 |

### 6.2 索引说明

- **`IX_Reservation_SeatId_TimeRange`** 是最关键的查询性能索引。预约提交时的时段冲突查询 SQL 类似：

```sql
SELECT COUNT(*) FROM [Reservation]
WHERE [SeatId] = @seatId
  AND [StartTime] < @endTime
  AND [EndTime] > @startTime
  AND [Status] IN (0, 1)  -- 待签到 或 使用中
```

这个复合索引使查询可以直接在索引中完成（覆盖索引），不需要回表。

- 其他索引都是针对高频查询设计的：体验账号查询自己的预约（StudentName）、管理端筛选状态（Status）、关联查询（SeatId）。

## 7. EF Core Fluent API 配置

### 7.1 AppDbContext

```csharp
// Data/AppDbContext.cs
public class AppDbContext : DbContext
{
    public AppDbContext(DbContextOptions<AppDbContext> options) : base(options) { }

    public DbSet<Seat> Seats => Set<Seat>();
    public DbSet<Reservation> Reservations => Set<Reservation>();
    public DbSet<Admin> Admins => Set<Admin>();

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        modelBuilder.ApplyConfiguration(new SeatConfiguration());
        modelBuilder.ApplyConfiguration(new ReservationConfiguration());
        modelBuilder.ApplyConfiguration(new AdminConfiguration());

        // 种子数据
        modelBuilder.Entity<Seat>().HasData(SeedData.GetSeats());
        modelBuilder.Entity<Reservation>().HasData(SeedData.GetReservations());
        modelBuilder.Entity<Admin>().HasData(SeedData.GetAdmins());
    }
}
```

               .IsRequired()

## 8. 状态流转与数据一致性

### 8.1 状态流转导致的数据变更

| 操作 | Reservation 变更 | Seat 变更 | 事务要求 |
|------|-----------------|-----------|----------|
| 提交预约 | 新增记录，Status=待签到 | Status←已预约 | 两个写入必须在一个事务中 |
| 取消预约 | Status←已取消 | Status←空闲 | 同上 |
| 确认签到 | Status←使用中 | Status←使用中 | 同上 |
| 释放座位 | Status←已完成 | Status←空闲 | 同上 |

### 8.2 Seat ↔ Reservation 状态联动矩阵

```
Seat.Status        Reservation.Status        是否允许
─────────────────────────────────────────────────────
空闲               （无关联预约记录）            可预约
已预约              待签到                      可取消（学生）、可签到（管理员）
使用中              使用中                      可释放（管理员）
空闲（释放后）      已完成（保留记录）            不可再操作，状态终止
空闲（取消后）      已取消（保留记录）            不可再操作，状态终止
```

### 8.3 不允许的数据状态组合

| 禁止的组合 | 说明 |
|-----------|------|
| Seat=空闲 且 Reservation=使用中 | 座位空闲但预约使用中 → 数据不一致 |
| Seat=已预约 且 Reservation=已取消/已完成 | 座位已预约但预约已结束 → 座位状态未释放 |
| Seat=使用中 且 Reservation=待签到 | 座位占用但预约未签到 → 不应该发生 |

### 8.4 事务实现方式

使用 `DbContext.Database.BeginTransaction()` 保证 Reservation 和 Seat 的更新原子性：

```csharp
// ReservationService.CheckIn 的事务实现
public void CheckIn(int reservationId)
{
    using var transaction = _context.Database.BeginTransaction();
    try
    {
        var reservation = _context.Reservations
            .Include(r => r.Seat)
            .First(r => r.Id == reservationId);

        if (reservation.Status != ReservationStatus.待签到)
            throw new BusinessException("当前预约状态不允许签到");

        reservation.Status = ReservationStatus.使用中;
        reservation.Seat.Status = SeatStatus.使用中;

        _context.SaveChanges();
        transaction.Commit();
    }
    catch
    {
        transaction.Rollback();
        throw;
    }
}
```

## 9. 种子数据设计

### 9.1 设计原则

- 种子数据是演示级——够用、好看、能覆盖全部状态
- 使用 `HasData()` 方法在 Migration 中自动插入

- 预约记录的 DateTime 使用相对值（相对于运行当天），保证始终看起来是"今天"的数据

## 10. 数据库迁移策略

### 10.1 迁移命令

```bash
# 初次创建迁移
dotnet ef migrations add InitialCreate

# 更新数据库
dotnet ef database update

# 后续修改实体后
dotnet ef migrations add AddXxxField
dotnet ef database update

# 回滚迁移
dotnet ef database update PreviousMigrationName
```

### 10.2 迁移流程

```
修改 Entity 类
    │
    ▼
dotnet ef migrations add 迁移名称    ← 生成迁移文件
    │
    ▼
审核迁移文件中的 Up() / Down() 方法  ← 检查生成的 SQL 是否正确
    │
    ▼
dotnet ef database update            ← 执行迁移
    │
    ▼
检查数据库表结构和种子数据            ← 确认无误
```

### 10.3 迁移原则

| 原则 | 说明 |
|------|------|
| **不手动改库** | 所有数据库变更通过 Migration 生成，绝不直接 SSMS 改表 |
| **迁移前备份** | 开发阶段不需要备份（LocalDB 数据不重要），但要有"出问题就 drop 重建"的心理准备 |
| **增量迁移** | 每次改实体建新迁移，不修改历史迁移文件（除非尚未提交） |
| **种子数据只增不改** | 已有种子数据可以追加，但不修改已有的 HasData 记录（EF Core 按主键匹配，修改会生成 UPDATE） |
| **开发阶段可重建** | 频繁改实体时，允许 `dotnet ef database drop` + `dotnet ef database update` 重建 |

### 10.4 迁移疑难处理

| 问题 | 处理方式 |
|------|----------|
| `dotnet ef` 命令找不到 | 安装工具 `dotnet tool install --global dotnet-ef` |
| 迁移报错"已有对象名" | 历史迁移和新实体冲突 → 删库重建或删除所有迁移重新 InitialCreate |
| 种子数据插入失败（主键冲突） | 确保 HasData 中的 Id 值与已有数据不冲突 |
| 外键导致删除失败 | `DeleteBehavior.NoAction`，不要在数据库直接 DELETE |

## 12. 数据库设计检查清单

### 12.1 建表前确认

- [ ] 是否只用 3 张表（Seat / Reservation / Admin），不新增？
- [ ] 字段类型是否匹配（string→nvarchar，bool→bit，DateTime→datetime2）？
- [ ] 主键是否统一为 Id（int, 自增）？
- [ ] 外键是否加了索引？
- [ ] 时段冲突查询的复合索引是否已建？
- [ ] 唯一约束是否覆盖了业务上不应重复的字段组合？
- [ ] 枚举是否统一用 int 存储，且默认值正确？

### 12.2 种子数据确认

- [ ] 14 个座位覆盖 2 个楼层、3 个区域，4 种不同的设施标签组合？
- [ ] 6 条预约记录覆盖全部 4 种状态？
- [ ] 同一座位在时间轴上不能有冲突的预约（Id=1 使用中 9-12，Id=4 已完成 14-17，不冲突）？
- [ ] 至少 3 个体验账号有预约记录？
- [ ] 管理员账号可用（admin / 123456）？

### 12.3 迁移后确认

- [ ] 数据库 3 张表已创建，列名、类型、约束与设计一致？
- [ ] 种子数据已插入，查询能返回预期结果？
- [ ] 唯一约束生效（尝试插入相同座位号应报错）？
- [ ] 外键约束生效（尝试删除有预约的座位应报错）？
- [ ] 复合索引已创建（`IX_Reservation_SeatId_TimeRange`）？
