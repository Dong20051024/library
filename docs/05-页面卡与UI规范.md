## PC-01 用户首页

| 项目 | 内容 |
|------|------|
| **页面名称** | 用户首页 |
| **页面角色** | 学生进入系统的第一站，承担"选哪个功能"的导航决策 |
| **页面目标** | 让学生快速理解"这是什么系统、我现在是谁、我能干什么" |
| **页面入口** | 浏览器打开 `/` 或 `/Home/Index`；体验账号切换后重定向至此 |
| **页面出口** | 点击导航或功能卡片进入座位列表页、我的预约页 |
| **最重要的信息** | 当前体验账号是谁（导航栏右上角显示"当前：学生1"）；系统是否正常运行（首页可正常打开） |
| **最重要的操作** | 切换体验账号（下拉选择）；点击"查看座位"进入核心流程 |
| **成功状态** | 页面展示系统名称、欢迎语，三个清晰的功能入口卡片（查看座位、我的预约、切换账号），导航栏显示当前账号 |
| **空状态** | 不涉及空状态（首页没有数据列表） |
| **异常提示** | 如果数据库连接失败或种子数据未初始化，首页应显示"系统初始化中，请稍后重试"的提示，而不是崩白页或异常堆栈 |
| **手机端布局** | 导航栏简化：左→系统名，右→当前账号（可点击切换）。功能入口采用 3 个纵向堆叠的卡片，占满屏幕宽度，间距 16px。页面标题用 22px 加粗。 |
| **后端关键点** | 体验账号列表由后端返回（种子数据中的 Students 表或配置列表）；首页不需要身份校验 |

## PC-02 座位列表页

| 项目 | 内容 |
|------|------|
| **页面名称** | 座位列表页 |
| **页面角色** | 系统核心入口页，学生判断"有没有空座"的第一现场 |
| **页面目标** | 让学生一眼看出哪些座位是空闲的，快速筛选到目标楼层/区域 |
| **页面入口** | 首页点击"查看座位"；座位详情页点击"返回列表" |
| **页面出口** | 点击空闲座位卡片进入座位详情页；点击顶部导航进入其他页面 |
| **最重要的信息** | 每个座位的当前状态（空闲/已预约/使用中）——用颜色区分，不看文字也知道 |
| **最重要的操作** | 点击空闲座位查看详情 |
| **成功状态** | 页面加载后展示所有座位，空闲（绿色卡片）、已预约（黄色卡片）、使用中（红色卡片）清晰可辨。每个座位显示座位号。顶部有楼层筛选下拉框，默认"全部楼层"。 |
| **空状态** | 如果没有任何座位数据（种子数据未导入），显示"暂未录入座位信息，请联系管理员"。如果某一楼层筛选后没有空闲座位，显示"当前楼层暂无空闲座位"。 |
| **异常提示** | 列表加载失败时显示"座位信息加载失败，请刷新页面重试"。 |
| **手机端布局** | 座位卡片采用 2 列网格，每个卡片占约 50% 宽度。卡片高度 100px，内部显示：座位号（顶部居中）、状态色条（底部）。筛选框固定在列表顶部，使用下拉选择器。不要做复杂的表格布局。 |
| **后端关键点** | 后端返回 Seat 列表 + 当前状态（实时计算或直接从 Seat.Status 读取）；支持按 Floor 参数筛选；需提供可选楼层列表（/api/seat/floors 或同接口附带） |

## PC-03 座位详情页

| 项目 | 内容 |
|------|------|
| **页面名称** | 座位详情页 |
| **页面角色** | 学生做预约决策的信息页面 |
| **页面目标** | 展示座位的全部信息，让学生决定"要不要约这个座位" |
| **页面入口** | 座位列表页点击空闲座位卡片 |
| **页面出口** | 点击"预约"按钮进入预约提交页；点击"返回列表"回到座位列表页 |

| **最重要的操作** | 点击"预约"按钮（仅空闲状态显示此按钮） |
| **成功状态** | 页面清晰展示：座位号（大号粗体）、楼层和区域、设施标签列表（用徽章样式）、当前状态（带颜色的状态标签）、"预约"按钮可用 |
| **空状态** | 不涉及（页面基于具体座位 ID 加载，如果 ID 不存在显示"该座位不存在"） |

| **后端关键点** | 根据 seatId 返回 Seat 完整信息（含设施标签列表）；预约按钮可见性由后端返回的 Seat.Status 控制；需同步返回当前预约时段信息（如果已预约） |

## PC-04 预约提交页

| 项目 | 内容 |
|------|------|
| **页面名称** | 预约提交页 |
| **页面角色** | 预约操作的核心表单页 |
| **页面目标** | 让学生选择日期和时段，成功提交预约 |
| **页面入口** | 座位详情页点击"预约"按钮 |
| **页面出口** | 提交成功 → 自动跳转到我的预约页；提交失败 → 停留在当前页显示错误信息 |
| **最重要的信息** | 所选座位的编号（让用户确认"我约的是这个座位"） |
| **最重要的操作** | 选择日期 → 选择起始时段 → 点击"确认预约" |
| **成功状态** | 提交成功后显示绿色成功提示（toast 或页面提示），然后 2 秒内自动跳转到我的预约页 |
| **空状态** | 不涉及 |
| **异常提示** | 时段冲突：显示"该座位在您选择的时间段已被预约，请选择其他时段或座位"。网络错误：显示"提交失败，请检查网络后重试"。预约时间不合理（如结束时间早于开始时间）：前端立即提示"结束时间不能早于开始时间"。 |
| **手机端布局** | 单列表单。从上至下：座位信息摘要（座位号 + 楼层区域，带返回图标）→ 日期选择器（点击弹出日期选择）→ 起始时间下拉 → 结束时间下拉 → "确认预约"按钮（全宽，蓝色，圆角 8px，高度 48px）。表单元素间距 16px。 |
| **后端关键点** | 提交时传递 seatId + studentName（当前体验账号） + startTime + endTime。后端需做时段重叠校验（同一座位、同一时段不重复预约）。成功返回预约记录 ID 和状态。 |

## PC-05 我的预约页

| 项目 | 内容 |
|------|------|
| **页面名称** | 我的预约页 |
| **页面角色** | 学生查看和管理自己所有预约记录的中心 |
| **页面目标** | 让用户知道自己约了哪些座位、当前状态是什么、能做什么操作 |
| **页面入口** | 首页点击"我的预约"；预约提交成功后自动跳转 |
| **页面出口** | 点击"取消预约"（执行后刷新列表）；点击导航进入其他页面 |
| **最重要的信息** | 每条预约记录的三个关键字段：座位号、时段、状态 |
| **最重要的操作** | 点击"取消预约"按钮（仅"待签到"状态显示） |
| **成功状态** | 列表展示当前账号的所有预约记录，按时间倒序排列（最新的在最上面）。每条记录清晰显示：座位号、楼层、时段、状态标签。状态标签颜色：待签到=蓝色、使用中=橙色、已完成=灰色、已取消=灰色（加删除线效果或淡化）。每个状态标签旁配一个短文字说明。 |
| **空状态** | 如果当前账号没有任何预约记录，显示空状态插图或图标 + 文字"暂无预约记录，快去预约座位吧" + "去选座"按钮引导到座位列表页。 |
| **异常提示** | 取消预约失败：显示"取消失败，请稍后重试"；加载失败：显示"预约记录加载失败，请刷新页面"。 |
| **手机端布局** | 单列列表，每个预约记录占一个卡片。卡片内容：座位号（左，加粗）+ 时段（左，次要信息）+ 状态标签（右对齐）。底部如果有"取消"按钮，单独一行在卡片最下方。卡片之间间距 12px。 |
| **后端关键点** | 根据当前体验账号（studentName）查询 Reservation 列表，按创建时间倒序。取消预约接口仅允许 Status="待签到"的记录执行取消，后端需要二次校验。取消后同步更新关联 Seat 的状态。 |

| **后端关键点** | 验证 Username + Password 是否匹配 Admin 表记录。登录成功后写入 Session（存储 AdminId, DisplayName）。密码本期明文比较（数据库不加密）。 |

## PC-07 预约管理页

| 项目 | 内容 |
|------|------|
| **页面名称** | 预约管理页 |
| **页面角色** | 管理员最核心的工作台——处理所有预约 |
| **页面目标** | 让管理员快速查看、筛选和处理预约（签到确认、释放座位） |
| **页面入口** | 登录成功后默认跳转至此；左侧菜单点击"预约管理" |
| **页面出口** | 点击左侧菜单进入其他管理端页面 |
| **最重要的信息** | 预约列表——每条记录的学生名、座位号、时段、状态 |
| **最重要的操作** | 对"待签到"记录点击"确认签到"；对"使用中"记录点击"释放座位" |
| **成功状态** | 列表加载正常，所有预约按创建时间倒序显示。状态筛选器位于列表上方，当前选中状态高亮。每条记录右侧有对应的操作按钮（根据状态不同显示不同的按钮）。签到或释放操作后有绿色成功提示，列表不刷新但操作行的状态立即更新。 |
| **空状态** | 如果没有预约数据，显示"暂无预约记录"（正常情况下不会有，种子数据会预置几条）。如果筛选后无匹配结果，显示"没有符合条件的预约记录"。 |
| **异常提示** | 签到失败：显示"签到失败，该预约状态可能已变更，请刷新后重试"。释放失败：显示"释放失败，请刷新后重试"。加载失败：显示"预约数据加载失败"。 |
| **桌面端布局** | 后台标准布局：顶部导航（左侧 logo，右侧管理员名 + 退出按钮）+ 左侧窄菜单 + 右侧主内容区。主内容区：顶部是筛选条件行（状态筛选按钮组 + 搜索框），下方是表格。表格列：学生名（120px）、座位号（100px）、时段（200px）、状态（100px）、操作（150px）。表格行高 52px，隔行浅灰背景。操作按钮图标+文字。 |
| **后端关键点** | 预约列表需支持按状态筛选（参数：status）。签到操作需同时更新 Reservation.Status 和 Seat.Status（事务保证）。释放操作同理。操作接口需幂等（重复点击不报错或提示"已处理"）。演示数据量小，不做分页。 |

# 第二部分：UI 规范与原型准备说明

---

## 1. 色彩规范

### 视觉主方向：**校园服务感**

> 整体感觉像校园官网或教务系统——可信、干净、不花哨，但比传统教务系统多一点年轻感。用户端偏暖，管理端偏理性。主色选用略带温度的"学院蓝"，呼应图书馆安静、专注的氛围。

### 主色板

| 用途 | 色值 | 色名 | 说明 |
|------|------|------|------|
| 主色 | `#2B5F8A` | 学院蓝 | 导航栏、主按钮、链接、重点标题 |
| 主色-浅 | `#E8F0F8` | 学院蓝浅色 | 卡片背景、表格隔行、选中状态 |
| 主色-深 | `#1A3D5C` | 学院蓝深色 | 按钮 hover、导航栏文字 |
| 辅助色 | `#D4A843` | 书卷金 | 强调标签、徽章、小面积高亮 |

## 3. 视觉方向说明

### 主方向：校园服务感

> 图书馆座位预约系统是一个工具型产品，不是娱乐产品。视觉设计目标是：**让用户觉得可信、顺手、不碍眼**。

**三个关键词：**
- **可信**：用色克制、排版整齐、状态清晰——学生打开页面就知道这个系统靠得住
- **顺手**：按钮在预期位置、操作反馈即时、信息层级分明——不需要思考"下一步点哪里"
- **不碍眼**：不做大色块、大阴影、大动效——让内容本身说话

### 用户端 vs 管理端差异

| 维度 | 用户端（校园服务感） | 管理端（现代后台感） |
|------|-------------------|-------------------|
| 基调 | 温暖、友好、引导性 | 理性、高效、数据感 |
| 主色使用 | 大面积主色（导航、卡片头部） | 主色用于关键操作和选中态 |
| 卡片感 | 明显（圆角 12px，微阴影） | 弱化（列表为主，减少装饰） |
| 排版 | 居中、留白多 | 左对齐、紧凑（信息密度高） |
| 动效 | 页面切换简单过渡 | 操作反馈（按钮状态变化） |

### 禁用风格

| ❌ 不要 | ✅ 要 |
|---------|------|
| 渐变背景、大范围渐变色 | 纯色块、干净背景 |
| 3D 效果、拟物化纹理 | 扁平设计、细微阴影 |
| 大面积深色/暗色模式（本期不做） | 浅色背景、高对比度文字 |
| 过度动效、弹跳动画 | 简单淡入、状态切换反馈 |
| 中文字体用衬线体（宋体等） | 无衬线中文字体 |
| 饱和度极高的颜色（荧光色） | 降低饱和度的学院色调 |

## 4. 页面氛围关键词与禁用风格

### 氛围关键词

```
用户端：   安静、明亮、有条理、引导清晰
管理端：   专业、克制、信息密集、操作高效
```

### 禁止进入的风格

```
❌ "极简主义白板"——什么都没有，学生不知道点哪里
❌ "酷炫 dashboard"——大面积深色、霓虹色、环形进度条
❌ "卡通可爱风"——圆角过大、过于活泼的图标和配色
❌ "传统教务系统灰"——灰色太多、无色彩倾向、死板
```

## 6. 响应式原则

### 用户端（移动端优先）

| 断点 | 目标设备 | 布局策略 |
|------|----------|----------|
| < 576px | 手机竖屏 | 单列布局，所有内容纵向堆叠。导航简化。卡片全宽。按钮全宽。 |
| 576px - 768px | 手机横屏/小平板 | 座位列表改为 2 列网格。表单保持单列。 |
| ≥ 992px | 桌面 | 保留移动端设计但放宽间距。座位列表可改为 3-4 列网格。最大内容宽度 960px，居中。 |

> 原则：**从手机竖屏开始设计**，然后逐步放大。不要在桌面端设计完再压缩到手机端。

### 管理端（桌面端优先）

| 断点 | 布局策略 |
|------|----------|
| ≥ 1200px | 标准后台布局（左侧菜单 + 右侧内容） |
| 768px - 1199px | 侧边栏折叠为汉堡菜单，展开时悬浮覆盖内容区 |
| < 768px | 表格显示核心列（隐藏次要列），操作按钮改为图标 |

> 原则：管理端的手机适配是"能用"，不是"好用"。优先保证在 1366px+ 屏幕上的操作效率。

## 7. 原型三层次说明

> 原型按三个精度层次递进。不是每个页面都需要全部三个层次。

| 层次 | 工具推荐 | 精度 | 做多少 | 用来干嘛 |
|------|----------|------|--------|----------|
| 低保真（Low-Fi） | 纸笔 / Excalidraw / Draw.io | 黑白线框，无颜色，简单矩形表示组件 | 每个页面画一个基本布局框 | 确认信息结构和页面跳转是否正确 |
| 中保真（Mid-Fi） | Figma / Sketch / Axure | 带颜色、实际文字内容、真实组件样式 | 核心页面（座位列表、预约提交、预约管理） | 评审交互流程和操作反馈 |
| 高保真（Hi-Fi） | Figma（像素级） | 与上线效果几乎一致，含真实数据、动效示意 | 仅 1-2 个关键页面（座位列表 + 预约管理）可选 | 最终视觉确认、演示截图 |

## 8. 哪些页面先做低保真，哪些直接做中保真

### 建议直接做中保真的页面（布局已明确、组件已定义）

| 页面 | 理由 |
|------|------|
| **座位列表页** | 核心页面，卡片网格布局的样式直接影响用户体验，需要颜色区分状态 |
| **预约提交页** | 表单元素多（日期、时间下拉、按钮），需要验证表单布局是否合理 |
| **我的预约页** | 状态标签多，需要验证不同状态的视觉区分度 |
| **预约管理页** | 表格 + 操作按钮 + 筛选器，需要验证后台操作的流畅性 |
| **管理员登录页** | 表单简单但需要体现品牌色，可以直接出最终视觉 |

### 建议先做低保真的页面（需要确认信息结构）

| 页面 | 理由 |
|------|------|
| **用户首页** | 需要确认功能入口的数量和排列方式（3 个卡片还是 4 个？） |
| **座位详情页** | 需要确认设施标签的展示方式（换行 vs 横向滚动？） |
| **座位管理页** | 需要确认新增/编辑的方式（弹窗 vs 行内编辑 vs 跳转新页面） |
| **统计页** | 需要确认展示哪些指标和图表类型 |

### 原型制作顺序建议

```
第一轮 → 座位列表页（中保真） + 预约提交页（中保真）
第二轮 → 预约管理页（中保真） + 管理员登录页（中保真）
第三轮 → 用户首页（低保真→转中保真） + 我的预约页（中保真） + 座位详情页（低保真）
第四轮 → 座位管理页（低保真） + 统计页（低保真，可选）
```

## 9. 原型评审时要重点看什么

### 评审清单（每页评审时逐项检查）

**信息结构**
- [ ] 用户能不能在 3 秒内理解"这个页面是干什么的"
- [ ] 最重要的信息是否在视觉焦点位置（不要求第一眼看到，但要求第二眼看到）
- [ ] 次要信息是否被适当弱化（字号、颜色、位置）

**操作流程**
- [ ] 核心操作按钮是否在预期位置（Fitts 定律：用户手指/鼠标自然停留区域）
- [ ] 操作反馈（成功/失败/进行中）是否有对应的 UI 状态
- [ ] 页面跳转是否符合用户预期（不跳转到意想不到的页面）

**移动端适配（用户端页面）**
- [ ] 手机上所有内容是否不需要缩放就能阅读
- [ ] 按钮是否足够大（至少 44px 高度）以便手指点击
- [ ] 表单输入框是否不会被键盘遮挡

**管理端操作区**
- [ ] 表格在 1366px 屏幕上是否需要横向滚动
- [ ] 操作按钮是否容易误触（靠近的删除和编辑按钮是否有足够间距）
- [ ] 筛选器是否在用户需要的位置（列表上方，而不是侧边）

**视觉一致性**
- [ ] 同一组件在不同页面是否使用同一套样式（按钮、输入框、标签）
- [ ] 颜色使用是否准确（语义色是否和规范一致）
- [ ] 间距系统是否一致（页面间相同层级元素的间距是否统一）
