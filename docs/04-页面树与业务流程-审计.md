## D. 建议优化项

### D.1 【建议】统一"8 MVP + 1 可选"的口径出现在文档开头

§1 的页面分布表将统计页计入"管理端 4 页"，容易让读者误以为统计页是确定的。建议在 §1 分布表中加一个脚注：

> 分布表快速总览，完整优先级见 §8。统计页为 P2（可选），实际 MVP 为 8 页。

或直接在分布表的管理端行改为"3 页（+1 页 P2 可选）"。

### D.2 【建议】体验账号切换在交付优先级中独立标注为 P0

§8 的交付优先级顺序表中，第一优先（P0）只有 4 个页面项。建议将体验账号切换显式列为第一优先条目：

```
第一优先 (P0)
─────────────
体验账号切换（全局导航组件）
座位列表页
预约提交页
管理员登录页
预约管理页
```

原因：体验账号切换是全局组件，影响所有用户端页面的导航布局和 Session 数据流，应该最先完成。

### D.3 【建议】预约提交失败分支补充到系统响应中

§4.1 第④步（预约提交页）的"系统响应"目前只描述了成功路径。建议拆分或补充：

> 提交成功：创建预约记录，跳转到我的预约页 → 预约状态=待签到，对应座位变为已预约
>
> 提交失败：停留在预约提交页，显示错误提示（如"该时段已被预约"）

### D.4 【建议】管理端双导航添加说明

§3.4 的管理端布局图展示了典型的 AdminLTE / Bootstrap Admin 模板双导航结构。建议加一行注释：

> 注意：管理端只有 3 个功能页面，双导航（顶部 + 侧边）是模板默认行为。侧边菜单作为主要导航，顶部导航提供快速切换和退出入口。

### D.5 【建议】统计页未实现时的访问策略

§3.2 中给 A4（统计页）加一行备注：

> 备注：如果 P2 统计页未实现，访问 `/Admin/Statistics` 路由应返回 404 或显示"功能开发中"提示；管理端菜单中的统计项应条件隐藏。

### D.6 【建议】体验账号切换丢失表单数据的取舍说明

在 §4.3 或 §2.3 的体验账号切换规则后增加一行：

> 设计取舍：切换体验账号后统一跳转回首页，不保留当前页面状态。这意味着如果用户在表单填写中切换账号，已填内容会丢失。当前简化实现，后续可根据需要改为"切换后刷新当前页面而非跳转首页"。
