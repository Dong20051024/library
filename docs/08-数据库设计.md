# 数据库设计 —— 图书馆座位预约系统

> **文档定位**：本文档是数据库建模的完整设计说明，包含实体定义、表结构、字段说明、状态设计、查询与统计支持、约束规则。开发者读完本文档应能直接编写 Migration 创建数据库。

## 1. 数据设计目标

| 目标 | 说明 |
|------|------|
| **支撑核心链路** | 座位列表 → 预约提交 → 我的预约 → 签到/释放，四个步骤的数据读写路径清晰 |
| **防止重复预约** | 同一座位同一日期同一时段，通过数据库查询+业务层双重校验保证不会被预约两次 |
| **管理端可查可筛** | 支持按日期、状态、区域、学生名四个维度查询预约记录 |
| **状态一致** | 座位状态和预约状态的联动更新在一个事务中完成，不会出现数据不一致 |
| **演示级种子数据** | 预置足够的数据覆盖全部状态，dotnet run 后直接可用 |

## 2. 核心实体清单

### 2.1 实体概览

前序文档（PRD-Lite §11、系统设计说明 §4.3）已明确 3 张核心数据表，本期不新增实体：

| # | 实体 | 中文名 | 核心角色 | 数据量预估 |
|---|------|--------|----------|-----------|
| 1 | **Seat** | 座位 | 座位的静态属性 + 动态状态 | 10~20 行（演示）/ 100~1000 行（生产） |
| 2 | **Reservation** | 预约记录 | 核心业务记录，链接学生、座位、时段、状态 | ~50 行（演示）/ 万级（生产） |
| 3 | **Admin** | 管理员 | 管理端身份认证 | 1~3 行 |

### 2.2 不建实体的数据

| 数据 | 处理方式 | 理由 |
|------|----------|------|
| 体验账号（学生1~学生5） | 代码常量 / 配置文件 | 没有注册流程、没有密码、没有个人信息，不需要建 User 表 |
| 区域（A区/B区/C区） | Seat.Area 字段 | 简单字符串标记，不做区域主数据管理 |
| 楼层 | Seat.Floor 字段 | 同上 |
| 设施标签 | Seat.Facilities 逗号分隔字段 | 本期不做设施的独立管理，标签只是展示属性 |
| 时段 | Reservation.StartTime / EndTime | 预约时间是自由范围，不是固定时段槽 |

**与默认 6 实体的对比：**

| 默认实体 | 在本设计中的处理 | 说明 |
|----------|----------------|------|
| User | 不建表 | 前序文档确认用 Session 存 StudentName 字符串即可 |
| SeatArea | 不建表 | 楼层+区域作为 Seat 的字符串字段（Floor, Area），不做主数据管理 |
| Seat | ✅ 保留 | 与 PRD-Lite 定义一致 |
| TimeSlot | 不建表 | 不设固定时段表，预约时间是自由范围的 StartTime~EndTime |
| Reservation | ✅ 保留 | 与 PRD-Lite 定义一致 |
| ReservationLog | 不建表 | 本期无操作日志需求，P2 统计直接从 Reservation 聚合 |

## 3. 关系说明

### 3.1 实体关系总图

```
 ┌──────────────┐                  ┌──────────────────┐
 │     Seat      │                  │   Reservation    │
 │──────────────│                  │──────────────────│
 │ PK: Id       │── 1 ───────── N ──→│ FK: SeatId      │
 │    Floor      │                  │    StudentName   │
 │    Area       │                  │    StartTime     │
 │    SeatNumber │                  │    EndTime       │
 │    Status     │                  │    Status        │
 │    Facilities │                  │    CreatedAt     │
 │    IsActive   │                  └──────────────────┘
 └──────────────┘

 ┌──────────────┐
 │    Admin      │  (独立，不与 Seat/Reservation 直接关联)
 │──────────────│
 │ PK: Id       │
 │    Username   │
 │    Password   │
 │    DisplayName│
 └──────────────┘
```

### 3.2 关系详细说明

| 关系 | 类型 | 说明 |
|------|------|------|
| Seat → Reservation | 1:N | 一个座位可以有多个预约记录（不同时间段的），但同一时间只有一条有效记录 |
| Reservation → Seat | N:1 | 每条预约对应一个座位 |
| Reservation → StudentName | 无外键 | 学生不是实体，StudentName 是直接在 Reservation 上的字符串字段 |
| Admin | 独立 | 不与 Seat 或 Reservation 关联，仅做登录认证 |

### 3.3 什么时候 Reservation 会有多条关联同一个 Seat

一个座位有 N 条预约记录是正常的——不同学生不同时间段的预约。例如：

```
Seat A-01:
  ├── Reservation #1 (学生1, 今天9:00-12:00, 使用中)   ← 当前有效
  ├── Reservation #4 (学生4, 昨天14:00-17:00, 已完成)  ← 历史
  └── ...更多历史记录
```

关键约束：同一座位在同一时间范围内，最多只有一条状态为"待签到"或"使用中"的记录。

## 4. 表结构设计

### 4.1 Seat（座位表）

存储座位的静态属性（在哪、有什么设施）和动态状态（当前是否被占用）。

| # | 字段名 | 类型（SQL） | 约束 | 默认值 | 说明 |
|---|--------|------------|------|--------|------|
| 1 | Id | `int` | PK, IDENTITY(1,1) | — | 主键，自增 |
| 2 | Floor | `nvarchar(20)` | NOT NULL | — | 楼层，如"3楼" |
| 3 | Area | `nvarchar(50)` | NOT NULL | — | 区域，如"A区"、"靠窗区"、"静音区" |
| 4 | SeatNumber | `nvarchar(20)` | NOT NULL | — | 座位编号，如"A-01" |
| 5 | Status | `int` | NOT NULL | 0 | 座位当前状态：0=空闲, 1=已预约, 2=使用中 |
| 6 | Facilities | `nvarchar(200)` | NOT NULL | '' | 设施标签，逗号分隔，如"有插座,靠窗,电脑位" |
| 7 | IsActive | `bit` | NOT NULL | 1 | 是否启用。禁用后座位不在用户端列表展示，不可预约 |

**唯一约束：** `(Floor, Area, SeatNumber)` 联合唯一。同一楼层同一区域内不能有两个座位编号相同。

**SQL DDL：**

```sql
CREATE TABLE [Seat] (
    [Id]          INT           IDENTITY(1,1) NOT NULL PRIMARY KEY,
    [Floor]       NVARCHAR(20)  NOT NULL,
    [Area]        NVARCHAR(50)  NOT NULL,
    [SeatNumber]  NVARCHAR(20)  NOT NULL,
    [Status]      INT           NOT NULL DEFAULT 0,
    [Facilities]  NVARCHAR(200) NOT NULL DEFAULT '',
    [IsActive]    BIT           NOT NULL DEFAULT 1,

    CONSTRAINT [UQ_Seat_Floor_Area_SeatNumber] UNIQUE ([Floor], [Area], [SeatNumber])
);
```

### 4.2 Reservation（预约记录表）

存储每一条预约的核心业务数据，是系统数据量最大、查询最频繁的表。

| # | 字段名 | 类型（SQL） | 约束 | 默认值 | 说明 |
|---|--------|------------|------|--------|------|
| 1 | Id | `int` | PK, IDENTITY(1,1) | — | 主键，自增 |
| 2 | SeatId | `int` | NOT NULL, FK → Seat(Id) | — | 外键，关联座位 |
| 3 | StudentName | `nvarchar(50)` | NOT NULL | — | 预约学生的体验账号名 |
| 4 | StartTime | `datetime2` | NOT NULL | — | 预约开始时间（年月日时分） |
| 5 | EndTime | `datetime2` | NOT NULL | — | 预约结束时间（年月日时分） |
| 6 | Status | `int` | NOT NULL | 0 | 预约状态：0=待签到, 1=使用中, 2=已完成, 3=已取消 |
| 7 | CreatedAt | `datetime2` | NOT NULL | GETDATE() | 记录创建时间 |

**外键约束：** `ON DELETE NO ACTION`（禁止删除仍有**有效预约记录（Status IN (0,1)，即待签到或使用中）**的座位。已完成和已取消的预约记录不阻塞删除）。

**索引：**

```sql
CREATE INDEX [IX_Reservation_SeatId] ON [Reservation]([SeatId]);
CREATE INDEX [IX_Reservation_StudentName] ON [Reservation]([StudentName]);
CREATE INDEX [IX_Reservation_Status] ON [Reservation]([Status]);
CREATE INDEX [IX_Reservation_SeatId_TimeRange]
    ON [Reservation]([SeatId], [StartTime], [EndTime]);
```

**SQL DDL：**

```sql
CREATE TABLE [Reservation] (
    [Id]          INT           IDENTITY(1,1) NOT NULL PRIMARY KEY,
    [SeatId]      INT           NOT NULL,
    [StudentName] NVARCHAR(50)  NOT NULL,
    [StartTime]   DATETIME2     NOT NULL,
    [EndTime]     DATETIME2     NOT NULL,
    [Status]      INT           NOT NULL DEFAULT 0,
    [CreatedAt]   DATETIME2     NOT NULL DEFAULT GETDATE(),

    CONSTRAINT [FK_Reservation_Seat]
        FOREIGN KEY ([SeatId]) REFERENCES [Seat]([Id])
        ON DELETE NO ACTION
);

CREATE INDEX [IX_Reservation_SeatId]
    ON [Reservation]([SeatId]);
CREATE INDEX [IX_Reservation_StudentName]
    ON [Reservation]([StudentName]);
CREATE INDEX [IX_Reservation_Status]
    ON [Reservation]([Status]);
CREATE INDEX [IX_Reservation_SeatId_TimeRange]
    ON [Reservation]([SeatId], [StartTime], [EndTime]);
```

### 4.3 Admin（管理员表）

存储管理员的登录凭据和显示名称。

| # | 字段名 | 类型（SQL） | 约束 | 默认值 | 说明 |
|---|--------|------------|------|--------|------|
| 1 | Id | `int` | PK, IDENTITY(1,1) | — | 主键，自增 |
| 2 | Username | `nvarchar(50)` | NOT NULL, UNIQUE | — | 登录账号 |
| 3 | Password | `nvarchar(100)` | NOT NULL | — | 密码（本期明文存储） |
| 4 | DisplayName | `nvarchar(50)` | NOT NULL | — | 显示名称，显示在管理端导航栏 |

**SQL DDL：**

```sql
CREATE TABLE [Admin] (
    [Id]          INT           IDENTITY(1,1) NOT NULL PRIMARY KEY,
    [Username]    NVARCHAR(50)  NOT NULL,
    [Password]    NVARCHAR(100) NOT NULL,
    [DisplayName] NVARCHAR(50)  NOT NULL,

    CONSTRAINT [UQ_Admin_Username] UNIQUE ([Username])
);
```

## 5. 关键字段说明

### 5.1 Seat.Facilities（设施标签）

**存储格式：** 逗号分隔的字符串，如 `"有插座,靠窗,电脑位"`

**为什么用逗号分隔而不是关联表：**
- 设施标签是一个纯展示属性，不需要单独管理（不做"新增设施标签"功能）
- 本期没有"按设施筛选座位"的需求
- 不需要对设施标签做统计分析
- 省去一张关联表和一次 JOIN 查询

**如果未来需要按设施筛选：** 可以用 `LIKE '%靠窗%'` 实现（数据量小时），或拆成多对多关联表。

**取值示例：**

| 取值 | 说明 |
|------|------|
| `""`（空字符串） | 无特殊设施 |
| `"有插座"` | 仅有一个设施 |
| `"有插座,靠窗"` | 多个设施，逗号分隔 |
| `"有插座,靠窗,电脑位"` | 三个设施 |

### 5.2 Reservation.StudentName（学生名）

**存储格式：** 体验账号的用户名字符串，如 `"学生1"`、`"学生2"`

**为什么不建 User 表 + 外键：**
- 体验账号不需要注册、不需要密码、没有个人资料
- 账号列表是静态数据，放在代码常量里维护
- 查询"某个学生的预约"直接用 `WHERE StudentName = '学生1'` 即可
- StudentName 上已建索引，查询性能足够

### 5.3 Reservation.StartTime / EndTime（预约时段）

**字段说明：**
- 两个字段都是 `datetime2` 类型，包含日期和时间
- 支持任意时间范围的预约（不限于整点、不限于固定时段槽）
- 例如：`2026-07-08 09:00:00` ~ `2026-07-08 12:00:00`

**时段范围校验规则：**
- EndTime 必须晚于 StartTime（前端+后端双重校验）
- 预约时长不能为 0（最小粒度 30 分钟，但技术上不强制）
- 跨天预约本期不禁止，但建议预约时长不超过 8 小时

### 5.4 Reservation.CreatedAt（创建时间）

- 数据类型 `datetime2`，由数据库 `DEFAULT GETDATE()` 自动填充
- 应用程序层不需要手动赋值
- 在"我的预约"中按此字段倒序排列

## 6. 状态字段设计

### 6.1 Seat.Status（座位状态）

| 值 | 枚举名 | 语义 | 能否预约 | 管理端操作 |
|----|--------|------|----------|-----------|
| 0 | 空闲 | 座位可用，任何人都可以预约 | ✅ 可以 | 无特殊操作 |
| 1 | 已预约 | 已被某学生预约，等待签到 | ❌ 不可以 | 可确认签到 |
| 2 | 使用中 | 学生已签到入座 | ❌ 不可以 | 可释放座位 |

**状态流转（座位视角）：**

```
      预约提交          签到
  空闲 ──────→ 已预约 ──────→ 使用中
    ↑                          │
    │ 取消预约 / 释放          │ 释放座位
    └──────────────────────────┘
```

**设计要点：**
- 座位没有"已完成"和"已取消"状态——这两个是预约的状态
- 座位状态是读密集型字段：每个座位列表页请求都要读取所有座位的状态
- 状态值以 int 存储，C# 中对应 `SeatStatus` 枚举

### 6.2 Reservation.Status（预约状态）

| 值 | 枚举名 | 语义 | 对应座位状态 | 可执行操作 |
|----|--------|------|-------------|-----------|
| 0 | 待签到 | 学生已提交预约，尚未到馆 | 已预约 (1) | 学生可取消；管理员可签到 |
| 1 | 使用中 | 学生已签到，正在使用座位 | 使用中 (2) | 管理员可释放座位 |
| 2 | 已完成 | 座位已释放，流程结束 | 空闲 (0) | 不可操作（只读） |
| 3 | 已取消 | 学生取消预约或超时未签到 | 空闲 (0) | 不可操作（只读） |

**状态流转（预约视角）：**

```
          管理员签到          管理员释放
  待签到 ────────→ 使用中 ────────→ 已完成
    │
    │ 学生取消
    ▼
  已取消
```

**设计要点：**
- 状态 0 和 1 是"有效"状态，代表预约正在进行中
- 状态 2 和 3 是"终态"，一旦进入不可再变更
- 时段冲突查询只检查状态为 0 和 1 的记录

### 6.3 座位 ↔ 预约状态联动对照

| 场景 | Seat.Status | Reservation.Status | 说明 |
|------|------------|-------------------|------|
| 初始状态 | 空闲 | （无关联预约） | 座位可用 |
| 预约刚提交 | 已预约 | 待签到 | 等待签到 |
| 签到完成 | 使用中 | 使用中 | 正在使用 |
| 释放座位 | 空闲 | 已完成 | 流程结束 |
| 取消预约 | 空闲 | 已取消 | 取消结束 |

### 6.4 不允许的状态组合

| 禁止的组合 | 原因 |
|-----------|------|
| Seat=空闲 且 Reservation=使用中 | 座位空闲但预约在使用 → 不一致 |
| Seat=已预约 且 Reservation=已完成 | 预约已结束但座位未释放 |
| Seat=已预约 且 Reservation=已取消 | 同上 |
| Seat=使用中 且 Reservation=待签到 | 座位被占但预约仍是待签到 |

## 7. 查询与筛选字段设计

### 7.1 用户端查询场景

| 页面 | 查询目标 | SQL 模式 | 涉及索引 |
|------|----------|----------|----------|
| 座位列表页 | 所有激活座位，可选按楼层筛选 | `SELECT * FROM Seat WHERE IsActive=1 [AND Floor=@floor] ORDER BY Floor, Area, SeatNumber` | —（少量数据，全表扫描即可） |
| 座位详情页 | 单个座位信息 | `SELECT * FROM Seat WHERE Id=@id` | PK_Seat |
| 预约提交页（时段冲突检查） | 同一座位同一时段是否有重叠的有效预约 | `SELECT COUNT(*) FROM Reservation WHERE SeatId=@sid AND StartTime<@end AND EndTime>@start AND Status IN (0,1)` | IX_Reservation_SeatId_TimeRange |
| 我的预约页 | 当前体验账号的所有预约，按创建时间倒序 | `SELECT * FROM Reservation WHERE StudentName=@name ORDER BY CreatedAt DESC` | IX_Reservation_StudentName |

### 7.2 管理端查询场景

| 页面 | 查询目标 | SQL 模式 | 涉及索引 |
|------|----------|----------|----------|
| 预约管理页 | 所有预约，可选按状态筛选 | `SELECT * FROM Reservation [WHERE Status=@status] ORDER BY CreatedAt DESC` | IX_Reservation_Status（按状态） |
| 预约管理页（按日期筛）* | 指定日期范围的预约 | `WHERE StartTime >= @from AND StartTime < @to` | IX_Reservation_SeatId_TimeRange |
| 座位管理页 | 所有座位 | `SELECT * FROM Seat ORDER BY Floor, Area, SeatNumber` | — |
| 统计页 | 近 7 天每天预约数量 | `SELECT CAST(StartTime AS DATE) AS [Date], COUNT(*) AS [Count] FROM Reservation WHERE StartTime >= @7daysAgo AND Status IN (0,1,2) GROUP BY CAST(StartTime AS DATE) ORDER BY [Date]` | IX_Reservation_SeatId_TimeRange |
| 统计页 | 当前座位利用率 | `SELECT Status, COUNT(*) FROM Seat GROUP BY Status` | — |

### 7.3 时段重叠查询——核心查询

时段重叠是系统最关键的查询，必须在每次创建预约前执行。两个时间段 `[a, b]` 和 `[c, d]` 重叠的条件是：`a < d AND c < b`。

```sql
-- 检查座位 @seatId 在 [@startTime, @endTime] 内是否已有有效预约
SELECT COUNT(*)
FROM [Reservation]
WHERE [SeatId] = @seatId
  AND [StartTime] < @endTime
  AND [EndTime] > @startTime
  AND [Status] IN (0, 1)    -- 只检查"待签到"和"使用中"的有效预约
```

如果 COUNT > 0，则拒绝创建预约。

**复合索引 `IX_Reservation_SeatId_TimeRange(SeatId, StartTime, EndTime)` 覆盖了此查询的所有 WHERE 条件，可以在索引层面完成过滤，不需要回表。**

### 7.4 管理端多维度查询

管理端预约管理页需要支持多维度组合筛选，SQL 模式：

```sql
SELECT r.*, s.Floor, s.Area, s.SeatNumber
FROM [Reservation] r
INNER JOIN [Seat] s ON r.[SeatId] = s.[Id]
WHERE 1=1
  [AND r.[Status] = @status]       -- 按状态筛
  [AND s.[Floor] = @floor]          -- 按楼层筛（区域通过楼层间接获得）
  [AND r.[StudentName] = @name]     -- 按学生名筛
  [AND r.[StartTime] >= @fromDate AND r.[StartTime] < @toDate]  -- 按日期范围筛
ORDER BY r.[CreatedAt] DESC
```

| 筛选维度 | SQL 对应 | 性能 |
|----------|----------|------|
| 状态 | `WHERE Status = @status` | 走 IX_Reservation_Status |
| 日期 | `WHERE StartTime >= @from AND StartTime < @to` | 走 IX_Reservation_SeatId_TimeRange |
| 区域（楼层） | 通过 JOIN Seat.Floor 筛选 | 走 IX_Reservation_SeatId 再 JOIN PK_Seat |
| 学生名 | `WHERE StudentName = @name` | 走 IX_Reservation_StudentName |

> \* 日期筛选为**可选扩展**：当前 MVP 管理端预约管理页仅按状态筛选（对应 Controller 和 Service 当前接口只接受 status 参数）。按日期范围筛选功能暂不实现，留待后续扩展。设计上保留 SQL 查询能力，但当前不暴露到 UI。
>
> 注意：按区域/楼层筛选需要 JOIN Seat 表。考虑到演示场景数据量很小（几十条到几百条），直接 JOIN 的性能完全可以接受。

## 8. 统计字段与统计口径

### 8.1 统计字段

所有统计数据均为 **实时计算**（运行期通过 SQL 聚合），不在数据库中存储冗余的统计字段。

| 指标 | 计算方式 | SQL 实现 | 用于页面 |
|------|----------|----------|----------|
| 每日预约数 | 按 Reservation.CreatedAt GROUP BY 日期 | `COUNT(*)` + `GROUP BY CAST(StartTime AS DATE)` | 统计页趋势图 |
| 座位利用率 | Seat 表中"使用中"座位占比 | `COUNT(CASE WHEN Status=2 THEN 1 END) * 100.0 / COUNT(*)` | 统计页利用率卡片 |
| 各状态座位数 | Seat 表按 Status GROUP BY | `SELECT Status, COUNT(*) FROM Seat GROUP BY Status` | 统计页分布图 |
| 个人预约数 | 按 StudentName 查询 | `SELECT COUNT(*) FROM Reservation WHERE StudentName=@name` | 我的预约页数量概览（可选） |

### 8.2 统计口径说明

| 统计项 | 口径 | 说明 |
|--------|------|------|
| 每日预约数 | 按预约的开始时间（StartTime）归属日期 | 例如 2026-07-08 09:00~12:00 的预约算在 07-08 |
| 座位利用率 | 当前时刻的快照值 | 显示的是"此时此刻有多少座位被占用" |
| 已完成预约是否计入统计 | ✅ 计入 | 已完成和已取消的预约也算历史数据 |
| 已取消预约是否计入趋势 | ❌ 可筛掉 | 统计趋势可以只算"有效预约"（状态 0,1,2），也可以全部计入；建议计入以便看到完整数据 |

## 9. 数据一致性与约束说明

### 9.1 数据完整性约束矩阵

| 约束类型 | 约束对象 | 实现方式 | 违反后果 |
|----------|----------|----------|----------|
| 主键约束 | 所有表的 Id | PK, IDENTITY | 不允许重复 |
| 唯一约束 | Seat.Floor + Area + SeatNumber | UNIQUE INDEX | 不允许重复座位号 |
| 唯一约束 | Admin.Username | UNIQUE INDEX | 不允许重复用户名 |
| 外键约束 | Reservation.SeatId → Seat.Id | FK, NO ACTION | 不允许指向不存在的座位；不允许删除有预约的座位 |
| NOT NULL | 所有业务字段 | DB 列约束 | 不允许空值 |
| 默认值 | Seat.Status = 0 | DEFAULT | 新座位默认空闲 |
| 默认值 | Seat.IsActive = 1 | DEFAULT | 新座位默认启用 |
| 默认值 | Reservation.Status = 0 | DEFAULT | 新预约默认待签到 |
| 默认值 | Reservation.CreatedAt = GETDATE() | DEFAULT | 自动记录创建时间 |

### 9.2 业务层一致性保证（应用层实现）

| 一致性要求 | 实现方式 |
|------------|----------|
| 同一座位同一时段不重复预约 | 业务层执行 `CreateReservation` 前先做时段重叠查询 |
| Seat ↔ Reservation 状态联动 | 使用 `DbContext.Database.BeginTransaction()` 事务提交两个表的更新 |
| 取消/签到/释放的幂等性 | 每个操作前校验当前状态是否符合预期 |
| 删除座位时禁止有未完成预约 | 业务层查询 Reservation 是否有关联的有效记录 |

### 9.3 事务边界

需要事务的操作（Seat 和 Reservation 两个表同步更新）：

| 操作 | 更新的表 | 事务范围 |
|------|----------|----------|
| 创建预约 | `Reservation`(INSERT) + `Seat`(UPDATE Status) | 同一事务 |
| 取消预约 | `Reservation`(UPDATE Status) + `Seat`(UPDATE Status) | 同一事务 |
| 确认签到 | `Reservation`(UPDATE Status) + `Seat`(UPDATE Status) | 同一事务 |
| 释放座位 | `Reservation`(UPDATE Status) + `Seat`(UPDATE Status) | 同一事务 |

### 9.4 并发处理说明

| 并发场景 | 问题 | 处理方式 |
|----------|------|----------|
| 两个学生同时预约同一个座位同一时段 | 两条重叠的预约都提交成功 | 使用事务 + 重叠查询中的 `UPDLOCK` 提示（SQL Server），或乐观并发：在 INSERT Reservation 后立即检查是否有时段冲突 |
| 管理员签到的同时学生取消 | 状态不一致 | 签到操作前校验 Reservation.Status 仍然是"待签到" |

本期对并发的处理原则：**不追求高并发下的完美正确性**（课堂项目，演示场景无人并发操作），但通过事务和状态校验避免明显的竞态问题。

### 11.2 当前假设

| 假设 | 影响范围 | 如果假设不成立 |

| 体验账号只有 5 个，用静态列表维护 | User 表可以不存在 | 如果需要支持真实用户注册/登录，需要新建 User 表，并将 Reservation.StudentName 改为 FK → User.Id |
