## C. 必改项

### C.1 【必改】补齐 F-A2（预约管理）的路由

**问题**：§6 中 F-A1、F-A5、F-A6 均有路由字段，F-A2 的验收标准表中缺少"路由"行。

**修改建议**：在 F-A2 验收标准表中增加一行：

| 路由 | `/Admin/Reservation` |

（与 §7 页面清单保持一致）

### C.2 【必改】修正座位状态机图中取消操作的箭头方向

**问题**：§12.1 座位状态机图中，取消操作的箭头从 [已取消] 指向 [已预约]，逻辑不通。正确方向应为：

```
[已预约] ──→ 取消操作 ──→ [已取消]
```

**修改建议**：将图中以下这段：

```
      │                  取消预约                        │ 管理员释放
      └──────────────── [已取消] ←──── (取消操作) ──────┘
```

修正为：

```
      │                  取消预约                        │ 管理员释放
      └─── (取消操作) ────→ [已取消] ──────────────────→ [空闲]
```

同时将 [已完成] 到 [空闲] 的回路补充完整（见 D.3）。

## D. 建议优化项

### D.1 【建议】将 F-A3/F-A4 整合为 F-A2 的子操作

既然 F-A3（签到确认）和 F-A4（释放座位）不是独立页面，而是 F-A2（预约管理）页面中的行内操作按钮，建议做如下调整之一：

**方案 A（推荐）**：将 F-A3 和 F-A4 的验收标准合并到 F-A2 中，删除独立的功能块。F-A2 的验收标准第 4、5 条已经覆盖了签到/释放的要求。

**方案 B**：保留 F-A3、F-A4 作为独立功能编号，但在每个功能块中加一行说明"此操作在预约管理页（F-A2）中执行，非独立页面"，消除"这看起来像三个页面"的误解。

### D.2 【建议】补齐"预约历史归档"的归属说明

原 MVP 确认中将 U8（预约历史归档）列为 P2 暂缓项，PRD Lite 中没有了。建议在 §14「从 MVP 中删减的功能」表中增加一行：

| 预约历史归档 | P2 暂缓项，本期 MVP 范围不包含，`我的预约`已满足基本查看需求 |

### D.3 【建议】补充 [已完成] → [空闲] 的状态机回路

座位状态机图中，释放操作后座位进入 [已完成] 状态但没有回到 [空闲]。虽然业务上可以理解为"已完成就是空闲的等价状态"，但从状态机严谨性出发，建议加上回路：

```
[已完成] ────→ [空闲]（自动恢复，可继续预约）
```

或在 [已完成] 旁边加注释："此状态等价于空闲，座位可重新被预约"。

### D.4 【建议】补充管理端列表页的空状态处理

在 F-A2（预约管理）和 F-A5（座位管理）的验收标准中各加一条：

- 列表无数据时显示"暂无预约记录" / "暂无座位数据"

### D.5 【建议】§10 验收标准简化为增量部分

§10 目前是 §5/§6 的重复汇总。建议精简为仅包含：

1. 跨功能的端到端验收（如"完整走通预约→签到→释放流程"）
2. 项目级验收（种子数据、仓库、README）
3. 与 §5/§6 不重叠的内容

功能级验收以 §5/§6 为单一权威来源。

### D.6 【建议】F-A5 删除座位的约束说明

在 F-A5 验收标准第 4 条后增加括号说明：

> 4. 支持删除座位（如果座位有关联的预约记录则禁止删除，**后端校验，前端同步禁用按钮**）

| 条件 | 结论 |
|------|------|
| 用户端和管理端功能是否都存在 | ✅ 是。用户端 7 项（F-U1~U7），管理端 6 项（F-A1~A6），完整覆盖 |
| 是否形成完整主流程 | ✅ 是。§8 的预约链路 + 管理端链路 + 完整时序覆盖了预约→签到→释放全生命周期 |

| 状态流转是否与项目题目匹配 | ✅ 是。座位 + 预约两套状态机，覆盖所有合法流转和禁止流转，与图书馆预约场景完全匹配 |

仅 2 个必改项，且都是文档内部一致性问题（缺少路由、状态机图箭头反了），不影响对业务逻辑和技术方案的理解。可以在进入下一阶段之前顺手修掉，或者先进入"页面树与业务流程"再回头修——**强烈建议先修再走**，尤其是 C.2（箭头方向反了），如果带着错误的状态机图进入设计阶段，可能导致后续产生误解。
