## 3. 做得好的地方

### 架构质量

- **Service 层与 Controller 层分离**：业务逻辑集中在 Service 中，Controller 仅做参数校验和视图调度
- **事务完整性**：预约提交（含冲突校验）和签到/释放操作均在事务中执行，保证 Seat 和 Reservation 状态一致性
- **自定义 BusinessException**：错误场景通过异常抛出，Controller 统一捕获并显示友好提示
- **AdminAuthorize 过滤器**：管理端权限控制集中在一处，无需每个 Action 单独写 Session 检查

### 数据完整性

- **时段冲突校验**：预约提交前检查同一座位在同一时段是否存在重叠的有效预约，使用 SQL 层重叠判断算法
- **取消预约权限校验**：校验操作人是否为预约的归属学生，阻止跨学生取消
- **删除座位保护**：有有效预约的座位无法删除

### 开发纪律

- 每个 Sprint 结束时 `dotnet build` 保持 0 error / 0 warning
- 自动化测试在首次运行时即捕获到路由冲突 Bug，验证了烟雾测试的有效性
- 文档与代码同步更新，17 份文档 + 任务板 + 数据库说明覆盖项目全生命周期

### 测试

- Playwright 烟雾测试覆盖 7 个关键场景，可在 15 秒内完成回归
- PowerShell 脚本烟雾测试零依赖（仅需 PowerShell 内置 cmdlet），适合快速检查服务是否运行
- Bug #001 的完整记录（发现→分析→修复→验证）可作为课程作业的典型 Debug 案例

## 7. 最终演示顺序建议

```
=== 场景一：核心链路（5 分钟）===
1. 首页 → 查看当前账号（学生1）和入口卡片
2. 切换到学生4 → 浏览座位列表（展示楼层筛选、状态颜色）
3. 点击空闲座位 A-03（带"靠窗"标签）→ 查看详情
4. 预约此座位 → 选择时段 → 提交 → 跳转"我的预约"（展示新预约，状态：待签到）
5. 管理员登录 → 预约管理页 → 找到新预约 → 签到（状态变更为"使用中"）
6. 释放座位 → 预约完成，座位恢复空闲

=== 场景二：管理端操作（3 分钟）===
7. 座位管理页 → 查看座位列表（含禁用、设施标签）
8. 新增座位 → 弹窗填写 → 保存 → 列表中出现新座位
9. 编辑座位 → 修改信息 → 保存
10. 删除座位 → 确认 → 删除（若有有效预约则提示无法删除）
11. 统计页 → 查看利用率卡片、分布条、趋势图

=== 场景三：权限验证（2 分钟）===
12. 切换到学生5 → 查看"我的预约"（空状态或仅有已取消记录）→ 无取消按钮
13. 切换到学生2 → 尝试取消 A-02 预约（待签到状态显示"取消预约"按钮）
```
