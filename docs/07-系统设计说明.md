## 1. 系统设计目标

| 目标 | 说明 |
|------|------|
| **分层清晰** | 每个类、每个方法都有明确归属，开发者不看文档也知道代码放哪 |
| **可单人交付** | 架构不复杂——不是微服务，不是 CQRS，不是 DDD——一个 .NET 8 MVC 项目走到底 |
| **可演示** | 种子数据预置后直接 dotnet run 就能演示核心流程 |
| **可扩展** | 本期不做的功能（统计页、数据导出等）预留了 Controller 和 Service 位置，不破坏现有分层 |

## 2. 技术栈说明

| 层 | 技术 | 版本 | 说明 |
|----|------|------|------|
| 运行时 | .NET | 8.0 | LTS 版本，课堂标准环境 |
| Web 框架 | ASP.NET Core MVC | 8.0 | 使用 Razor 视图引擎，不启用 Razor Pages |
| ORM | Entity Framework Core | 8.0 | Code-First 模式，迁移工具管理数据库 |
| 数据库 | SQL Server LocalDB | 随 VS2022 安装 | 本地开发数据库，无需额外安装 |
| 前端框架 | Bootstrap | 5.x | 通过 CDN 或 LibMan 引入 |
| 自定义样式 | 独立 CSS 文件 | — | `wwwroot/css/libseat.css`，完全迁移自原型 |
| 版本控制 | Git + GitHub/Gitee | — | 单体仓库，单分支开发 |

**不做：** 没有前端构建工具（Webpack/Vite）、没有 TypeScript、没有 Docker、没有反向代理。

**前端交互原则：** 核心操作（提交表单、签到、释放）走传统表单 POST + 页面刷新；辅助交互（下拉切换、弹窗确认、Toast 提示）用原生 JS + Bootstrap JS 组件，不引入 Vue/React 等框架。

### 目录设计要点

1. **Controllers/Admin/ 子目录**：不使用 Area 注册方式（这个项目管理端页面少，Area 反而增加复杂度）。直接在 `Controllers/Admin/` 下建控制器，通过 Route 属性区分命名空间。
2. **Models/Entities 和 Models/ViewModels 分离**：Entity 是数据库表映射，ViewModel 是视图专用数据结构。两者绝不混用。
3. **Services 层必须定义接口**：虽然这个项目不会有多实现切换（不会换数据库），但接口定义是分层纪律——Controller 只依赖接口，不依赖实现类。
4. **Data/Configurations/ 使用 Fluent API**：不用 Data Annotation 写在 Entity 类上，保持 Entity 干净。

## 4. 分层说明

### 分层总览

```
用户请求 (HTTP)
    │
    ▼
┌─────────────────────────────────────────────────┐
│  Controller 层                                    │
│  职责：接收请求 → 调用 Service → 组装 ViewModel →  │
│        返回 View / Redirect / JsonResult           │
│  不做的：不写业务逻辑、不直接操作 DbContext          │
└────────────────────┬────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────┐
│  Service 层                                       │
│  职责：业务规则（状态机、时段校验）、数据编排          │
│  做法：调用 DataAccess 层（DbContext）获取/写入数据   │
│  不做的：不处理 HTTP 上下文、不引用 HttpContext     │
└────────────────────┬────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────┐
│  DataAccess 层 (Data/ + Models/Entities/)        │
│  职责：EF Core DbContext、实体映射、数据库迁移       │
│  做法：定义 DbSet、OnModelCreating、Fluent API     │
│  不做的：不包含业务逻辑                              │
└─────────────────────────────────────────────────┘
```

### 4.1 Controller 层

```
namespace LibrarySeatSystem.Web.Controllers;
```

**职责清单：**

| 职责 | 示例 |
|------|------|
| 参数解析 | 从 RouteData、QueryString、Form 中取值传给 Service |
| 调用 Service | `_seatService.GetSeatList(floor)` |
| ViewModel 组装 | new ListViewModel { Seats = result, Floors = floors } |
| 视图返回 | `return View(viewModel)` 或 `return RedirectToAction(...)` |
| 错误处理 | try-catch 后返回 View 并设置错误提示 |
| Session 读取/写入 | 管理员登录写 Session，体验账号切换写 Session |
| 未登录拦截 | `[AdminOnlyFilter]` 检查 Session，无则跳转登录页 |

**约束：**

- Controller Action 中**不允许**出现 `using var db = new AppDbContext()` 或 `_context.Seats.ToList()` 这样的直接数据库操作。必须通过 Service 调用。
- Controller Action 中**允许**出现 ViewModel 组装代码（new ViewModel { ... }），但 ViewModel 内的数据来源必须是 Service 返回的。
- Controller 的构造方法只接收 Service 接口注入，不接收 DbContext 注入。

### 4.2 Service 层

```
namespace LibrarySeatSystem.Web.Services;
```

**职责清单：**

| 职责 | 示例 |
|------|------|
| 业务规则校验 | 创建预约时校验时段重叠、校验座位状态是否为"空闲" |
| 状态流转控制 | 确认签到时同时更新 Reservation.Status 和 Seat.Status（事务） |
| 数据查询封装 | `GetSeatList(floor)` 返回带状态的座位列表 |
| 数据写入 | `CreateReservation(...)` 写入 Reservation 表并更新 Seat 表 |
| 不允许的操作拦截 | 学生不能取消"使用中"的预约；不能删除有预约记录的座位 |

**约束：**

- Service 之间可以互相调用。例如 `ReservationService.CreateReservation` 需要查询座位状态时，应调用 `SeatService` 而不是直接查 `DbContext`。
- Service 方法抛出业务异常（如 `InvalidOperationException("时段冲突")`），由 Controller 层捕获并处理。
- Service 方法的返回值是 Entity 或 DTO（简易的 `record` 或 `class`），不是 ViewModel。ViewModel 组装在 Controller 层完成。

### 4.3 DataAccess 层

```
namespace LibrarySeatSystem.Web.Data;
```

**职责清单：**

| 职责 | 说明 |
|------|------|
| DbContext 定义 | 包含 3 个 DbSet：`DbSet<Seat>`、`DbSet<Reservation>`、`DbSet<Admin>` |
| Entity 映射 | Fluent API 配置表名、字段类型、索引、外键关系 |
| 数据库迁移 | `dotnet ef migrations add` / `dotnet ef database update` |
| 种子数据 | `OnModelCreating` 中通过 `HasData()` 预置演示数据 |

**Entity 定义原则：**

- Entity = 数据表映射，字段名、类型与数据库列一一对应。
- Entity 上可以写导航属性（`public Seat Seat { get; set; }`），但不在 Entity 上写业务方法。
- 枚举字段在数据库中以 int 存储，在 C# 中以 enum 类型使用。

```csharp
// Models/Entities/Seat.cs — 正确示例
public class Seat
{
    public int Id { get; set; }
    public string Floor { get; set; } = string.Empty;
    public string Area { get; set; } = string.Empty;
    public string SeatNumber { get; set; } = string.Empty;
    public SeatStatus Status { get; set; } = SeatStatus.空闲;
    public string Facilities { get; set; } = string.Empty;   // 逗号分隔
    public bool IsActive { get; set; } = true;

    // 导航属性
    public ICollection<Reservation> Reservations { get; set; } = new List<Reservation>();
}
```

### 4.4 ViewModel 层

```
namespace LibrarySeatSystem.Web.Models.ViewModels;
```

**定位：** ViewModel 是"为视图定制的一张表"。它和 Entity 的区别：

| | Entity | ViewModel |
|--|--------|-----------|
| 对应什么 | 数据库表的一行记录 | 页面上的一个展示单元 |
| 可嵌套 | 通常不嵌套（靠导航属性） | 可以嵌套（列表、下拉选项、分页信息） |
| 可包含列表 | 导航属性是 ICollection | 直接是 `List<SeatItem>` |
| 可包含计算字段 | 不行，除非 NotMapped | 可以，状态标签 CSS class、格式化后的时段字符串 |
| 可包含 UI 状态 | 不行 | 可以，比如 `IsCurrentUserReserved` |

**示例：**

```csharp
// Models/ViewModels/Seat/ListViewModel.cs
public class ListViewModel
{
    public List<SeatItem> Seats { get; set; } = new();
    public string? CurrentFloor { get; set; }
    public List<string> Floors { get; set; } = new();
    public string? ErrorMessage { get; set; }   // 异常提示
}

public class SeatItem
{
    public int Id { get; set; }
    public string SeatNumber { get; set; } = string.Empty;
    public string Floor { get; set; } = string.Empty;
    public string Area { get; set; } = string.Empty;
    public string StatusText { get; set; } = string.Empty;     // "空闲"
    public string StatusCssClass { get; set; } = string.Empty; // "ls-badge-free"
    public string Facilities { get; set; } = string.Empty;
}
```

### 4.5 Enum 层

枚举在 `Models/Enums/` 中单独定义，不写在 Entity 类文件里。

```csharp
// Models/Enums/SeatStatus.cs
public enum SeatStatus
{
    空闲 = 0,
    已预约 = 1,
    使用中 = 2
}

// Models/Enums/ReservationStatus.cs
public enum ReservationStatus
{
    待签到 = 0,
    使用中 = 1,
    已完成 = 2,
    已取消 = 3
}
```

### 各层依赖方向

```
Controller → Service (接口) → Service (实现) → DataAccess (DbContext)
    │                            │
    ▼                            ▼
ViewModel                      Entity (通过 Service 返回)
```

- Controller 知道 Service（接口）、知道 ViewModel
- Service 知道 Entity、知道 DbContext
- Service 之间知道彼此（互相调用）
- ViewModel 只知道自己的字段（不依赖 Entity）
- Entity 不知道 Controller、不知道 Service、不知道 ViewModel

## 7. 主链路工程落点

### 7.1 预约座位链路（用户端核心链路）

```
首页 → 列表 → 详情 → 提交 → 我的预约
```

| 步骤 | 代码落点 | 关键逻辑 |
|------|----------|----------|
| ① 用户访问首页 | `HomeController.Index()` → `Home/Index.cshtml` | 从配置或常量加载体验账号列表 → 放入 ViewModel |
| ② 查看座位列表 | `SeatController.List(string? floor)` → `SeatService.GetSeatList(floor)` → `Views/Seat/List.cshtml` | Service 查询时关联当前预约状态，空闲/Occupied 从 Seat.Status 读取 |
| ③ 点击空闲座位 | `SeatController.Detail(int id)` → `SeatService.GetSeatDetail(id)` → `Views/Seat/Detail.cshtml` | 校验 seatId 是否存在，不存在返回 NotFound 视图 |
| ④ 提交预约 | `ReservationController.Create(CreateViewModel model)` [POST] → `ReservationService.CreateReservation(seatId, studentName, startTime, endTime)` | 时段重叠校验 → 写入 Reservation → 更新 Seat.Status=已预约 → 事务提交 |
| ⑤ 我的预约 | `ReservationController.Mine()` → `ReservationService.GetMyReservations(studentName)` → `Views/Reservation/Mine.cshtml` | 按 StudentName 查询，按时间倒序返回 |

### 7.2 取消预约链路

| 步骤 | 代码落点 | 关键逻辑 |
|------|----------|----------|
| ① 点击取消 | `ReservationController.Cancel(int id)` [POST] | Service 校验当前状态是"待签到"才允许取消 |
| ② 取消执行 | `ReservationService.CancelReservation(id)` | 更新 Reservation.Status=已取消 → 更新 Seat.Status=空闲 → 事务提交 |

### 7.3 签到与释放链路（管理端核心链路）

```
登录 → 预约管理 → 签到 / 释放
```

| 步骤 | 代码落点 | 关键逻辑 |
|------|----------|----------|
| ① 管理员登录 | `Admin.LoginController.Index(LoginViewModel)` [POST] → `AdminService.Validate(username, password)` | 比较明文密码 → 写入 Session (AdminId, DisplayName) |
| ② 查看预约管理 | `Admin.ReservationController.Index(ReservationStatus? status)` → `ReservationService.GetAllReservations(status)` | 支持按状态筛选，返回列表 |
| ③ 确认签到 | `Admin.ReservationController.CheckIn(int id)` [POST] → `ReservationService.CheckIn(id)` | 事务：Reservation: 待签到→使用中, Seat: 已预约→使用中 |
| ④ 释放座位 | `Admin.ReservationController.Release(int id)` [POST] → `ReservationService.Release(id)` | 事务：Reservation: 使用中→已完成, Seat: 使用中→空闲 |

### 7.4 状态流转代码实现要点

每次状态变更都涉及**两个表的联动更新**，必须在一个事务中完成：

```csharp
// ReservationService.CheckIn 的业务逻辑伪代码
public void CheckIn(int reservationId)
{
    using var transaction = _context.Database.BeginTransaction();
    try
    {
        var reservation = _context.Reservations
            .Include(r => r.Seat)
            .FirstOrDefault(r => r.Id == reservationId)
            ?? throw new NotFoundException("预约记录不存在");

        // 状态校验：只有"待签到"可以签到
        if (reservation.Status != ReservationStatus.待签到)
            throw new InvalidOperationException("当前状态不允许签到");

        // 更新预约状态
        reservation.Status = ReservationStatus.使用中;

        // 更新座位状态
        reservation.Seat.Status = SeatStatus.使用中;

        _context.SaveChanges();
        transaction.Commit();
    }
    catch
    {
        transaction.Rollback();
        throw;
    }
}
```

## 8. 异常处理原则

### 8.1 异常分层

| 层 | 异常类型 | 处理方式 |
|----|----------|----------|
| Service | `NotFoundException`（自定义） | 数据不存在时抛出 |
| Service | `InvalidOperationException` | 业务规则不满足时抛出（时段冲突、状态不允许） |
| Service | `DbUpdateException` | EF 写入失败（外键约束、并发冲突） |
| Controller | 所有 Service 异常 | try-catch 捕获，设置 ViewModel.ErrorMessage，返回 View |
| Controller | 未捕获异常 | 全局异常中间件 → 显示友好错误页面 |

### 8.2 自定义业务异常

```csharp
// 可以定义业务异常类，与系统异常区分
public class BusinessException : Exception
{
    public string UserMessage { get; }
    public BusinessException(string userMessage) : base(userMessage)
    {
        UserMessage = userMessage;
    }
}
```

### 8.3 Controller 层异常处理模式

```csharp
public IActionResult Create(CreateViewModel model)
{
    if (!ModelState.IsValid)
        return View(model);

    try
    {
        _reservationService.CreateReservation(
            model.SeatId,
            HttpContext.Session.GetString("StudentName") ?? "学生1",
            model.StartTime,
            model.EndTime
        );
        return RedirectToAction(nameof(Mine));
    }
    catch (BusinessException ex)
    {
        model.ErrorMessage = ex.UserMessage;
        return View(model);  // 停留当前页，显示错误
    }
}
```

### 8.4 全局异常处理

在 `Program.cs` 中配置：

```csharp
if (!app.Environment.IsDevelopment())
{
    app.UseExceptionHandler("/Home/Error");
}
```

在生产环境下，500 错误统一跳转到 `Views/Shared/Error.cshtml`，显示"系统繁忙，请稍后重试"，不暴露异常堆栈。

## 9. Session / Cookie 使用原则

### 9.1 体验账号（用户端）

| 项目 | 说明 |
|------|------|
| 存储方式 | `HttpContext.Session.SetString("StudentName", "学生1")` |
| 默认值 | 第一次访问时如果没有 Session，默认设为"学生1" |
| 切换操作 | 用户端导航栏的下拉切换 → POST 到 `/Home/SwitchAccount?name=学生2` → 更新 Session → 重定向首页 |
| 有效期 | 浏览器关闭即失效（Session 模式） |
| Cookie 名 | 使用 ASP.NET Core 默认 `.AspNetCore.Session` |
| 安全性 | 本机演示，不要求加密；但 Cookie 应设为 `SameSite=Lax` |

### 9.2 管理员登录（管理端）

| 项目 | 说明 |
|------|------|
| 存储方式 | `HttpContext.Session.SetInt32("AdminId", adminId)` + `SetString("AdminDisplayName", name)` |
| 登录操作 | POST 到 `/Admin/Login` → 校验通过 → 写入 Session → 跳转预约管理页 |
| 登出操作 | POST 到 `/Admin/Logout` → `Session.Clear()` → 跳转登录页 |
| 有效期 | 浏览器关闭即失效，或 30 分钟无操作过期 |
| 拦截方式 | 自定义 `AdminAuthorizeAttribute`（ActionFilter）检查 Session 中是否有 AdminId |

### 9.3 AdminAuthorizeAttribute 实现要点

```csharp
public class AdminAuthorizeAttribute : ActionFilterAttribute
{
    public override void OnActionExecuting(ActionExecutingContext context)
    {
        var adminId = context.HttpContext.Session.GetInt32("AdminId");
        if (adminId == null)
        {
            context.Result = new RedirectToActionResult("Index", "Login", new { area = "Admin" });
        }
    }
}
```

在管理端 Controller 上使用：

```csharp
[Route("Admin/[controller]")]
[AdminAuthorize]
public class ReservationController : Controller
{
    // 所有 Action 都需要登录
}

// 登录页本身不加过滤器
[Route("Admin/[controller]")]
public class LoginController : Controller
{
    public IActionResult Index() => View();
}
```

### 9.4 不使用 Cookie 的场景

| 场景 | 原因 | 替代方案 |
|------|------|----------|
| 记住登录状态 | 本期没有"记住我"需求 | 每次关闭浏览器自动登出 |
| 用户端登录态持久化 | 体验账号不需要持久化 | Session 即可 |
| 跨页面数据传递 | 不使用 Cookie 传数据 | 使用 ViewModel + 路由参数 |

| 我的预约 + 取消预约 | 全部实现 | 按账号查询 + 取消操作（仅待签到可取消） |

| 预约管理（签到/释放） | 全部实现 | 状态流转 + 事务控制 |

### 10.2 可选实现（P2 → 1 页）

| 功能 | 实现原则 |
|------|----------|
| 统计页 | 先实现后端接口（`StatisticsService`），视时间决定是否做前端图表展示。如果时间不够，可以在侧边栏保留菜单项但跳转后显示"开发中" |

| 功能 | 排除理由 |
|------|----------|

| 多馆区 | 本期只有一个图书馆 |
| API 接口（RESTful） | 本系统是传统 MVC，不是前后端分离 |
| 单元测试/集成测试 | 课堂项目，手工测试即可 |
| HTTPS | LocalDB 本地演示，HTTP 足够 |
| 数据备份/恢复 | 不需要 |

### 10.4 实现顺序建议

按页面交付优先级分三轮开发：

```
第一轮（P0 链路闭环，约 50% 工作量）
├── 搭建项目骨架 + 数据库 + 种子数据
├── 座位列表页（核心入口）
├── 预约提交页（核心操作）
├── 管理员登录页（管理端入口）
├── 预约管理页（签到/释放）
└── 确认：一个完整的"空闲→已预约→使用中→空闲"链路能走通

第二轮（P1 完善，约 35% 工作量）
├── 用户首页（体验账号切换）
├── 座位详情页
├── 我的预约页 + 取消预约
├── 座位管理页（增删改）
└── 确认：所有 8 页可访问，无空白页

第三轮（P2 收尾 + 打磨，约 15% 工作量）
├── 统计页（如果时间充裕）
├── 错误提示统一检查
├── 响应式适配调整
└── 最终测试 + 提交仓库
```
