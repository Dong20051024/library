## C. 必改项

### C.1 【必改】体验账号切换（U12）需在进入 PRD Lite 前锁定实现方向

**问题**：U12 被标记为 P0（核心基础），这是正确的判断。但它的交互方式和实现机制被完全推迟到 PRD Lite 阶段确定。对于一个 P0 功能来说，这引入了一个前置风险——PRD Lite 如果发现 U12 的实现方案影响页面路由结构或数据库设计，会导致前期工作返工。

**修改建议**：

在本文件中至少明确以下 3 个决策之一，作为推荐方案进入 PRD Lite：

**推荐方案——导航栏下拉切换器 + Session 存储**

- 交互：页面顶部导航栏右侧放置一个下拉选择器，列出 3-5 个预置学生姓名（学号），点击即切换当前用户
- 实现：切换时发 POST 请求到 `/Account/Switch`，后端将 `UserId` 写入 Session；所有页面从 Session 读取当前用户
- 种子数据：预置 3-5 个体验账号（Student 表），各自带几条示例预约记录
- 不需要登录页、不需要密码、不需要注册

**为什么必须在此时确定**：这个方案决定了 Session 中间件的启用、全局用户上下文的注入方式、以及数据库 Student 表的设计——都是 PRD Lite 和后续开发的前置条件。

在 §5 或 §8 中增加上述方案描述。

### C.2 【必改】明确 A5（签到）和 A6（释放）的操作位置和交互

**问题**：A5（手动签到确认）和 A6（手动释放座位）是 P0 功能，构成预约状态流转的关键环节。当前它们只在功能列表中列了个名字，没有说明：

- 操作按钮放在哪个页面？
- 是列表行内操作还是详情页操作？
- 操作后页面如何反馈？

**修改建议**：

在 §5 管理端「预约管理」的功能描述中明确：

> 预约管理（含签到/释放）：展示全部预约记录列表，支持按状态筛选。每条"待签到"记录提供"签到入座"按钮，点击后状态变为"使用中"；每条"使用中"记录提供"释放座位"按钮，点击后状态变为"已释放"，座位恢复空闲。操作后页面刷新或局部更新状态。

这个描述也解决了立项单审计中 C.2 的遗留问题（签到操作的功能归属），并给出了具体的交互方式。

## D. 建议优化项

### D.1 【建议】预约时段方案前置选定

§8 将"固定时段 vs. 自由起止"列为 PRD Lite 待定事项。建议在本文档中选定一个默认方案，避免 PRD Lite 阶段犹豫不决：

- **推荐方案**：自由选择起止时间（精确到 30 分钟），系统最小预约时长 1 小时，最大 8 小时
- 理由：自由起止比固定时段更灵活，更真实地反映图书馆使用场景；实现上只需要 `StartTime` + `EndTime` 两个 DateTime 字段，外加时长验证逻辑

如果不放心，可以加一句"PRD Lite 可根据实际情况调整"作为安全网。

### D.2 【建议】补充"管理员在岗"假设

在 §10 风险与假设区域（可以新增一个假设条目）：

> 假设：管理员在工作时间（如 8:00-22:00）在岗，学生到馆后可找到管理员完成签到确认。

### D.3 【建议】修正 P0 工作量标注

将 §4 的 P0 工作量从"约 20%"调整为"约 30%"，更贴近实际情况。如果觉得 20% 更符合课时预期，至少加一条注释说明 P0 包含 Session 管理和全局筛选逻辑，开发时有额外成本。

### D.4 【建议】MVP 验收准则交叉检查

在 §5 MVP 范围确认后增加一条备注：

> 立项单 §9 成功标准第 9 条（管理员统计页）因 A4 暂缓调整为 P2 不纳入 MVP 验收。其余验收标准仍适用。MVP 阶段结束后补充分阶段验收检查表。

### D.5 【建议】U11 优先级调整说明

如果 U11（开放时间/公告）确实实现成本很低（一个静态页面 + 管理员可编辑），可以考虑从 P3 删除改为 P2 暂缓，作为"M3 提前完成后顺手加上"的候选。当前标记为删除也是合理的，此条仅作为技术建议供参考。

## E. 是否可以进入下一步"PRD Lite"

| 条件 | 结论 |
|------|------|
| 是否做了真的需求分析而非改写立项单 | ✅ **是**。场景定义、用户关注点、候选功能池扩展、优先级矩阵——这些是真正的分析工作 |
| 是否有明确的功能优先级矩阵 | ✅ **有**。P0-P3 分类，每个功能有评分理由，覆盖用户端和管理端 |
| 是否有明确的 MVP 范围确认 | ✅ **有**。P0+P1 = MVP，P2 可选，P3 排除。范围清单清晰 |
| 是否有删减项 / 暂缓项 | ✅ **有**。§6 排除项 + §7 删减项/暂缓项，理由充分，触发条件明确 |
| 需求评审结论是否成立 | ⚠️ **基本成立，但有 2 个 P0 功能的实现路径需要现在就收口，不能拖到 PRD Lite** |
| 最终范围是否适合单人课堂项目 | ✅ **适合**。8 页 MVP + 1 页可选统计页，对一个 .NET + EF Core 课堂项目来说是合理负载 |

**结论：建议在以下条件满足后进入 PRD Lite：**

1. **必改项 C.1**：体验账号切换（U12）锁定实现方案（推荐导航栏下拉 + Session 存储），写入文档
2. **必改项 C.2**：签到/释放操作（A5/A6）明确交互位置和方式，写入文档
3. 建议顺手补齐 **D.1**（预约时段默认方案）和 **D.4**（MVP 验收准则交叉检查），改动量小

这两个必改项的本质是同一个问题：**P0 功能的实现路径不应带着"待定"标签进入下一阶段**。在进入 PRD Lite（页面级设计）之前，P0 功能至少应该有一个明确的实现方向，否则 PRD Lite 会出现参照系漂移。
