## C. 必改项

### C.1 【必改】统一管理端视图存放路径与控制器注册方式

**问题**：文档 §3 目录结构将管理端视图放在 `Areas/Admin/Views/` 下，但控制器在 `Controllers/Admin/` 下并声明"不使用 Area 注册方式"。这会导致 Razor 视图发现引擎找不到管理端视图。

**根因**：ASP.NET Core MVC 的默认视图发现规则是：
- 对于 `Controllers/Admin/ReservationController`（非 Area，使用 `[Route("Admin/[controller]")]`），视图搜索路径为：
  1. `Views/Admin/Reservation/Index.cshtml`
  2. `Views/Shared/Index.cshtml`
- **不会**搜索 `Areas/Admin/Views/Reservation/Index.cshtml`

**修复方案（二选一）：**

| 方案 | 改动 | 推荐度 |
|------|------|--------|
| **方案 A（推荐）**：将管理端视图从 `Areas/Admin/Views/` 移到 `Views/Admin/` | 删除 `Areas/` 目录，管理端视图移至 `Views/Admin/Login/`、`Views/Admin/Reservation/`、`Views/Admin/Seat/`、`Views/Admin/Statistics/`。这样做最简洁，与"不使用 Area"的决策完全一致 | ⭐ |
| **方案 B**：完全采用 Area 方式 | 将控制器移入 `Areas/Admin/Controllers/`（而不是 `Controllers/Admin/`），在 `Program.cs` 中注册 Area 路由。这样做工程上也可行，但文档需要增加 Area 注册代码，略微增加复杂度 | |

**推荐方案 A**，理由：与文档已有的"管理端页面少，Area 增加复杂度"的决策一致；目录从 2 层（Areas/Admin/Views/）改为 1 层（Views/Admin/），更扁平。

**对应的目录结构调整**：

```
Before (文档中的)              After (方案 A)
────────────────────          ────────────────
Areas/Admin/Views/             Views/Admin/
  Login/Index.cshtml   →         Login/Index.cshtml
  Reservation/Index.cshtml →     Reservation/Index.cshtml
  Seat/Index.cshtml      →       Seat/Index.cshtml
  Statistics/Index.cshtml →      Statistics/Index.cshtml
```

## D. 建议优化项

### D.1 【建议】在 §6 映射表中补充 SwitchAccount 入口

在 §6 的映射表中增加一行：

| 切换体验账号 | `/Home/SwitchAccount` | `HomeController` | `SwitchAccount(string name)` [POST] | → Redirect to Index | P1 |

同时注明该 Action 接收 POST 请求，更新 Session 中的 StudentName 后重定向到首页。

### D.2 【建议】指定体验账号列表的全局传递方案

在 §9.1 中增加一段说明，指定账号列表如何传递到所有用户端页面。

**推荐方案：注入 Service + 部分视图**

在 `_Layout.cshtml` 中通过 `@inject ISeatService SeatService`（或独立的 `IAccountService`）获取账号列表，渲染到导航栏的下拉组件中。不需要在每个 Controller Action 中传递。

如果觉得注入 Service 到视图不够"纯 MVC"，可以选方案二：

**备选方案：ViewBag + 基础 Controller**

定义一个 `BaseController`，在其 `OnActionExecuting` 中加载账号列表到 `ViewBag.Accounts`，布局页从 ViewBag 读取。

无论选哪个方案，建议在文档中写清楚。

### D.3 【建议】明确不分页的数据量上限

在 §10.3 "不做分页"后面补充：

> 数据量预期：种子数据预置不超过 50 条记录，课堂演示场景下全量加载不会造成性能问题。如果实际使用中数据量超过 200 条，建议在座位管理页和预约管理页增加 Skip/Take 分页。

### D.4 【建议】体验账号模块独立为 AccountService

§5 将体验账号管理挂在 `SeatService` 下（备注"SeatService（账号列表部分）"），逻辑上不属于座位模块。建议：

- 新增 `IAccountService` / `AccountService`（可放在 `Services/` 下）
- 职责：获取体验账号列表、切换账号（写入 Session 由 Controller 处理）
- 这样和 `SeatService` 的职责分离，未来如果要加真实用户注册也更容易扩展

或者更轻量的做法：在 `Models/SeedData/SeedData.cs` 中定义静态账号列表，HomeController 直接读取，不经过 Service 层（够简单就不需要 Service）。

### D.5 【建议】统一过滤器命名

将 §4.1 中的 `[AdminOnlyFilter]` 和 §9.3 中的 `[AdminAuthorize]` 统一为同一个名称。推荐 `[AdminAuthorize]`，因为它更清楚地表达了"授权"语义。
