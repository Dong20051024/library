### 5. Controller、Service、DataAccess 分工是否合理

**结论：✅ 分工合理，但有 1 处与 docs/07 §4.2 的设计原则有偏差。**

#### 正确遵循的分工

| 原则 | docs/07 定义 | docs/09 执行情况 | 结果 |
|------|-------------|----------------|------|
| Controller 处理 HTTP 上下文 | 读 Session、写 Session | `Index()`: 读 Session 初始化 StudentName<br>`Create` [POST]: 读 Session 取 StudentName<br>`Login` [POST]: 写 Session AdminId | ✅ |
| Controller 调用 Service 接口 | 只依赖接口 | 所有 Controller 注入的都是 `ISeatService`、`IReservationService`、`IAdminService` | ✅ |
| Controller 不操作 DbContext | 不出现 `_context.Seats.ToList()` | 没有 Controller 直接操作 DbContext | ✅ |
| Service 处理业务规则 | 状态机、校验、数据编排 | `CreateReservation` 做 3 项校验 + 事务写入<br>`CheckIn` 校验状态 + 事务更新 | ✅ |
| Service 不使用 HttpContext | 不引用 HttpContext | 所有 Service 方法都不引用 HttpContext | ✅ |
| Service 使用事务 | 使用 `DbContext.Database.BeginTransaction()` | 所有写操作（Create/Cancel/CheckIn/Release）都用了事务 | ✅ |
| DataAccess 只做读写 | — | Service 中通过 `_context` 操作数据，没有额外的 DataAccess 层 | ✅ |

#### 与设计原则有偏差的地方

**偏差：Service 方法返回 ViewModel，而非 Entity 或 DTO**

docs/07 §4.2 明确写：

> "Service 方法的返回值是 Entity 或 DTO（简易的 record 或 class），**不是 ViewModel**。ViewModel 组装在 Controller 层完成。"

但本文档中：

| Service 方法 | 返回类型 | 属于 ViewModel？ |
|-------------|---------|-----------------|
| `GetSeatList(string? floor)` | `ListViewModel` | ✅ 是 ViewModel |
| `GetSeatDetail(int id)` | `DetailViewModel?` | ✅ 是 ViewModel |
| `GetMyReservations(string studentName)` | `MineViewModel` | ✅ 是 ViewModel |
| `GetAllReservations(ReservationStatus?)` | `AdminReservationViewModel` | ✅ 是 ViewModel |

**这好还是不好？**

从课堂项目看，**这其实是可以接受的权衡**：
- 避免了 Controller 中多余的一层映射代码
- Service 直接返回 Controller 需要的数据，Controller 变得很薄
- 项目只有 3 张表、8 页，ViewModel 变化很少

但如果严格按照分层原则，应该：

1. Service 返回 `List<SeatDto>`（DTO）
2. Controller 将 `List<SeatDto>` 映射为 `ListViewModel`

**建议**：在 docs/09 §9.1 补足项中加一条，说明这是"有意的简化选择"，不是"忘了分层原则"。

## D. 建议优化项

### D.1 【建议】在 §9.1 补足项表中增加"Service 返回 ViewModel 是有意简化"

### D.2 【建议】补上 `Admin/Logout` 的代码实现

在 §5.2 末尾增加：

```csharp
// POST: /Admin/Logout
[HttpPost]
[AdminAuthorize]
[ValidateAntiForgeryToken]

{
    HttpContext.Session.Clear();
    return RedirectToAction("Index", "Login");
}
```

### D.3 【建议】统一 `CreatedAt` 的处理方式

二选一：
- 保留 C# 赋值 `CreatedAt = DateTime.Now` → 在 docs/08 中更新 §5.4 的表述为"应用程序层也会赋值，DB 默认值作为后备"
- 或者遵循 docs/08 §5.4，不在 C# 端赋值 → 从 `CreateReservation` 代码中删掉 `CreatedAt = DateTime.Now`

推荐选第二种（代码更少，与 docs/08 一致）。

### D.4 【建议】给 SwitchAccount 加上 `[ValidateAntiForgeryToken]`

在 §4.2 的 SwitchAccount 代码上加一行：

```csharp

```

### D.5 【建议】删掉多余的 `.AsQueryable()` 调用

在 §5.3 `GetAllReservations` 中：

```csharp
// 改前
var query = _context.Reservations.Include(r => r.Seat).AsQueryable();

// 改后
var query = _context.Reservations.Include(r => r.Seat);
```

`DbSet<T>` 已实现 `IQueryable<T>`，`AsQueryable()` 是多余的。

### D.6 【建议】将 `GetSeatList` 的两个查询合并为一个

在 §2.3 `GetSeatList` 中，当前逻辑是：

```csharp
// 查询 1：获取座位列表
var seats = query.OrderBy(...).Select(...).ToList();

// 查询 2：获取楼层去重列表
Floors = _context.Seats.Where(s => s.IsActive).Select(s => s.Floor).Distinct()...ToList();
```

可以简化：用第一个查询的结果 `.Select(s => s.Floor).Distinct()`。这不影响性能（数据量 < 50 行），但少一次数据库查询更干净。
