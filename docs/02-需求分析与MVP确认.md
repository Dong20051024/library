# 需求分析与 MVP 确认 —— 图书馆座位预约系统

> 本文档为 PRD 之前的收束动作。不涉及真实用户访谈，采用"课堂替代调研"方式完成：基于校园场景合理假设 + 参考同类预约系统常见能力 + 按课堂项目工作量收束范围。

---

## 1. 目标用户与核心场景

### 用户角色

| 角色 | 典型身份 | 计算机熟练度 | 使用频率 |
|------|----------|-------------|----------|
| 学生（读者） | 在校本科生/研究生 | 日常使用手机和电脑 | 考试周每天，平时每周 2-3 次 |
| 图书馆管理员 | 图书馆工作人员/勤工助学学生 | 基础办公软件水平 | 每天多次，工作时间持续使用 |

### 核心场景

**场景 A — 学生预约座位（高频、核心）**

> 考试周期间，小张早上 7:30 起床，打开电脑登录图书馆预约系统，看到三楼靠窗区还有一个空闲座位，选了 8:00-12:00 的时段提交预约。到馆后找前台管理员确认签到，入座学习。中午离开时告知管理员，座位释放。

**场景 B — 学生查看和取消预约（中频）**

> 小张上午的课临时调到下午，之前约的座位去不了了。他打开"我的预约"，找到那条"待签到"的记录，点取消。座位立刻被释放，其他人可以预约。

**场景 C — 管理员管理座位（中频、管理端）**

> 管理员小李发现四楼靠墙的一张椅子坏了，登录后台把该座位状态标记为"禁用"。又在三楼新增了 5 个带插座的新座位，方便学生给笔记本电脑充电。

**场景 D — 管理员查看预约数据（低频、管理端）**

> 月底了，馆长想看看座位利用率到底怎么样。管理员小李登录后台打开统计页，看到过去一周的平均利用率是 67%，周四下午最高，截图发给了馆长。

## 2. 用户最关心的问题

### 学生最关心的问题（按关注度排序）

| # | 问题 | 我们的回应 |
|---|------|-----------|
| 1 | 我怎么知道现在哪个座位是空的？ | 座位列表页用颜色区分状态，空闲的绿色显示 |
| 2 | 约了座位之后别人会不会抢走？ | 预约成功后座位锁定，别人不能再约这个时段 |
| 3 | 约错了或者去不了了怎么办？ | 在"我的预约"里可以直接取消，操作简单 |
| 4 | 这个座位有没有插座？是不是靠窗？ | 座位详情页展示设施标签（插座/靠窗/电脑位等） |
| 5 | 要不要注册账号？太麻烦了 | 不需要注册，直接用体验账号切换，一键换人 |

### 管理员最关心的问题

| # | 问题 | 我们的回应 |
|---|------|-----------|
| 1 | 我怎么确认谁约了哪个座位？ | 预约管理列表展示所有预约记录，支持按状态和时间筛选 |
| 2 | 座位坏了/新增座位怎么处理？ | 座位管理页支持增删改，操作后前台实时更新 |
| 3 | 座位够不够用？什么时候最挤？ | 统计页展示预约趋势图和座位利用率 |
| 4 | 登录方不方便？ | 固定账号 + Session，记住登录状态 |

## 4. 功能优先级矩阵

### 优先级分层说明

| 层级 | 标签 | 含义 | 课堂工作量 |
|------|------|------|-----------|
| P0 | 核心基础 | 系统能跑起来的前提，缺了它项目不成立 | 约 20% |
| P1 | 主体业务 | 用户和管理员的核心操作链路 | 约 50% |
| P2 | 体验提升 | 有了更好，没有也能用 | 约 20% |
| P3 | 暂缓 | 本期不做，留给后续迭代 | 0% |

### 用户端优先级

| 编号 | 功能 | 优先级 | 理由 |
|------|------|--------|------|
| U12 | 体验账号切换 | **P0** | 没有用户识别就无法区分"我的预约"，系统跑不通 |
| U2 | 座位列表 | **P0** | 核心入口，学生第一眼看到的东西 |
| U4 | 预约提交 | **P0** | 系统最核心的操作，没有它就不叫预约系统 |
| U5 | 我的预约 | **P1** | 用户查看自己预约的唯一入口 |
| U6 | 取消预约 | **P1** | 用户体验的关键闭环，不能约了就不能取消 |
| U1 | 首页 | **P1** | 系统门面，引导进入各功能 |
| U3 | 座位详情 | **P1** | 提供座位详细信息，辅助决策 |
| U8 | 预约历史归档 | **P2** | 方便回顾，但刚上线时数据不多，需求不急 |
| U10 | 座位地图模式 | **P2** | 可视化效果好，但列表模式也能用；开发成本较高 |
| U9 | 收藏常用座位 | **P3** | 锦上添花，第一期没必要 |
| U11 | 开放时间/公告 | **P3** | 属于内容管理范畴，偏离核心预约链路 |
| U7 | 学生自助签到 | **P3** | 涉及更复杂的权限和状态校验，暂由管理员代劳 |

### 管理端优先级

| 编号 | 功能 | 优先级 | 理由 |
|------|------|--------|------|
| A1 | 管理员登录 | **P0** | 管理端入口，进不去后台什么都干不了 |
| A2 | 预约管理（含列表+筛选） | **P0** | 管理员最核心的日常工作界面 |
| A5 | 手动签到确认 | **P0** | 预约流程闭环的关键一环，缺了它状态流转卡住 |
| A6 | 手动释放座位 | **P0** | 同上，状态必须能走完"空闲→预约→使用→释放" |
| A3 | 座位管理（增删改） | **P1** | 管理员配置座位的基础功能，但不是每天用 |
| A4 | 数据统计 | **P2** | 可见即可得，但非核心操作链路的一部分 |
| A7 | 按日期查看报表 | **P2** | 统计功能的延伸，统一放在统计页实现 |
| A8 | 导出预约数据 | **P3** | 刚上线数据量不大，导出需求不急 |
| A9 | 管理员账号管理 | **P3** | 只有 1 个管理员账号，本期不需要管理多个 |
| A10 | 操作日志 | **P3** | 安全性延伸，课堂项目暂不需要 |

## 5. MVP 范围确认

### MVP 边界

> 经过优先级筛选，MVP 只包含 **P0 + P1** 的功能。P2 视时间决定是否纳入，P3 全部砍掉。

### MVP 功能清单

| 模块 | 包含功能 | 对应页面 |
|------|----------|----------|
| 用户端 | 首页、座位列表（含楼层/区域筛选）、座位详情、预约提交、我的预约、取消预约、体验账号切换 | 5 页（首页 + 座位列表 + 座位详情 + 预约提交 + 我的预约） |
| 管理端 | 管理员登录、预约管理（列表 + 状态筛选）、座位管理（增删改）、手动签到确认、手动释放座位 | 3 页（登录页 + 预约管理 + 座位管理） |
| 基础 | 项目搭建、EF Core + LocalDB 连通、种子数据、Bootstrap 布局、Git 初始化 | 不单独成页 |

### 统计页放到哪个优先级？（特别说明）

数据统计（A4）放在了 **P2（体验提升）**。原因是：

- 统计功能不影响核心预约链路的正常运转
- 课堂项目的演示重点通常是"能不能约座位"，而不是"图表好不好看"
- 如果时间充裕（M3 阶段提前完成），可以加上统计页
- 如果时间紧张，统计页作为**第一个可砍的功能**

**建议**：按"能做就做，做不了不丢分"的态度对待统计页。先确保 P0 和 P1 全部稳定跑通，再考虑统计页。

## 6. 本期明确不做的功能

> 以下功能经过评审已排除在 MVP 之外，但不代表"永远不做"，而是"本期不做"。列在这里防止开发过程中反复纠结。

### 技术上可以做、但本期主动排除的

| 功能 | 排除理由 | 未来可能纳入 |
|------|----------|-------------|
| 学生自助签到（扫码/输入码） | 签到流程简化，由管理员代劳 | 二期如果有自助签到机或二维码可以加 |
| 座位地图可视化 | 列表模式够用，地图模式开发成本高 | 二期如果前端能力提升可以加 |
| 收藏常用座位 | 预约操作本身不复杂，收藏需求不高 | 视用户反馈决定 |
| 导出预约数据 | 刚上线数据量小，截图也能汇报 | 后台数据多了之后加 |

### 立项单已声明不做的（复述确认）

| 功能 | 确认排除 |
|------|---------|
| 支付 / 押金 / 罚款 | ✅ 排除 |
| 扫码签到 / 扫码抢座 | ✅ 排除（签到走管理员手动确认） |
| 消息推送 | ✅ 排除 |
| 真实用户注册 / 密码找回 | ✅ 排除（用体验账号切换代替） |
| 多馆区联动 | ✅ 排除（本期只有一个馆） |
| 微信小程序 / App | ✅ 排除（只做 Web 版） |
| 实时座位占用检测（IoT） | ✅ 排除 |
| 黑名单 / 信用分 | ✅ 排除 |
| 第三方 OAuth 登录 | ✅ 排除 |

## 7. 需求评审结论

### 评审方式

课堂替代评审：基于立项单 + 场景假设 + 优先级矩阵，单人在此完成评审记录。

### 保留项（纳入 MVP）

| 编号 | 功能 | 保留理由 |
|------|------|----------|
| U1-U6, U12 | 全部用户端 P0/P1 功能 | 核心预约链路的完整闭环，缺一不可 |
| A1-A3, A5-A6 | 管理端登录、预约管理、签到确认、座位管理、释放座位 | 管理端支撑业务运转的基本能力 |

### 删减项（从立项单候选列表中直接移除）

| 编号 | 功能 | 删减理由 |
|------|------|----------|
| U7 | 学生自助签到 | 签到由管理员代劳；学生自助签到需要更复杂的身份验证机制，本期不做 |
| U9 | 收藏常用座位 | 功能独立但非必须，首次使用没有收藏数据，体验价值低 |
| U10 | 座位地图模式 | 列表 + 状态颜色已经可以清晰展示座位信息；地图涉及 Canvas 或 SVG，复杂度高 |
| U11 | 开放时间/公告 | 属于内容管理，不是预约系统的核心功能 |
| A8 | 导出预约数据 | 没有真实运营数据时导出功能无法验证 |
| A9 | 管理员账号管理 | 本期只有 1 个管理员，不需要管理功能 |
| A10 | 操作日志 | 课堂项目不需要审计追溯 |

### 暂缓项（本期保留但不保证交付，列在 MVP 目标之外）

| 编号 | 功能 | 暂缓理由 | 触发条件 |
|------|------|----------|----------|
| A4/A7 | 数据统计（含按日期查看） | 不影响核心链路，开发成本较高；需要图表库集成 | M3 提前完成后纳入 |
| U8 | 预约历史归档 | 有"我的预约"可以看到近期记录，归档不是刚需 | MVP 完成后剩余时间 |

## 8. 进入 PRD Lite 前的最终口径

### 一句话总结

> 做一个图书馆座位预约 Web 系统，学生能查座位、约座位、取消预约；管理员能管座位、管预约、确认签到。不做注册、不做支付、不做扫码、不做 App。

### MVP 最终范围速览

| 维度 | 内容 |
|------|------|
| **用户端页面** | 首页、座位列表、座位详情、预约提交、我的预约（共 5 页） |
| **管理端页面** | 管理员登录、预约管理、座位管理（共 3 页） |
| **核心数据表** | 座位表（Seat）、预约表（Reservation）、管理员表（Admin） |
| **用户识别** | 体验账号切换，SQL 种子数据预置 3-5 个体验账号 |
| **管理员登录** | 固定账号 + Session/Cookie |
| **签到方式** | 管理员后台手动确认（将"待签到"→"使用中"） |
| **统计页** | 暂缓，时间充裕再加 |
| **页面总数** | 8 页（P0 + P1）或 9 页（+ 统计页 P2，视时间） |

### PRD Lite 待明确事项

本文档完成后，下一步（PRD Lite）需要明确以下细节：

1. 每个页面的具体字段和信息结构
2. 数据库表结构和字段设计
3. 预约时段如何定义（固定时段 vs. 自由选择起止时间）
4. 体验账号切换的交互方式（下拉选择？点击切换？）
5. 座位状态的详细转换规则（状态机）
6. 管理员签到/释放的操作流程和 UI 位置
