### 3. 表之间关系是否合理

**结论：✅ 合理，依赖方向与业务一致。**

| 关系 | 类型 | 评语 |
|------|------|------|
| Seat → Reservation | 1:N (PK → FK) | 正确。一个座位可以有 N 条历史预约，同一时间最多 1 条有效 |

**唯一约束 `(Floor, Area, SeatNumber)`**：正确。同一楼层同一区域内不会有两个同编号的座位。

## D. 建议优化项

### D.1 【建议】补充利用率的第二种口径

**位置**：§8.1 座位利用率指标

**现状**：只算 Status=2（使用中）的座位占比。

**建议**：增加第二个指标"座位占用率"：

| 指标 | 计算方式 | 说明 |
|------|----------|------|
| 座位利用率（当前） | 使用中 / 总数 × 100% | 当前有多少人在实际使用 |
| 座位占用率（建议加） | （使用中 + 已预约）/ 总数 × 100% | 当前有多少座位已被占用（含已预约未签到） |

或者至少加一句说明：

> 利用率只统计"使用中"的座位（即当前有人入座）。"已预约"的座位不纳入分子，因为预约状态可能在短时间内变更（取消或过期未签到）。

### D.2 【建议】在 Reservation 表补充 IX_Reservation_StudentName_CreatedAt 复合索引

**位置**：§4.2 索引设计

**现状**：
```sql
CREATE INDEX [IX_Reservation_StudentName] ON [Reservation]([StudentName]);
```

**建议**：改为覆盖索引，把 `CreatedAt DESC` 包含进来，避免排序：
```sql
CREATE INDEX [IX_Reservation_StudentName_CreatedAt]
    ON [Reservation]([StudentName], [CreatedAt] DESC);
```

这样"我的预约"页面的查询 `WHERE StudentName=@name ORDER BY CreatedAt DESC` 可以在索引层面完成排序，不需要内存排序。

**注意**：这不是性能问题（课堂项目 < 50 条纪录不需要优化），而是"如果被问到为什么没有这个索引"时的防御性设计。

### D.3 【建议】明确 `CreatedAt` 的 EF Core 实现方式

**位置**：§4.2 的 `CreatedAt` 字段定义

**现状**：写 `DEFAULT GETDATE()`

**建议**：补充一句 EF Core 中的映射方式：

```csharp
// Fluent API 配置
entity.Property(e => e.CreatedAt)
    .HasDefaultValueSql("GETDATE()");
```

或者：

```csharp
entity.Property(e => e.CreatedAt)
    .ValueGeneratedOnAdd();
```

两者的区别：
- `HasDefaultValueSql("GETDATE()")`：数据库在 INSERT 时自动填充，EF Core Insert 时赋任意值会被数据库默认值覆盖
- `ValueGeneratedOnAdd()`：EF Core 在 INSERT 时由数据库生成，但 .NET 端也能读回

推荐用 `HasDefaultValueSql("GETDATE()")`，让数据库行为显式化。

### D.4 【建议】在 §7.2 的管理端查询部分加一句数据量说明

**位置**：§7.2 或 §7.4

**现状**：管理端预约管理页支持按 Floor 筛选（需要 JOIN Seat），以及多维度组合筛选，但现有索引不一定能覆盖组合筛选。

**建议**：加一条说明：

> 管理端多维度组合筛选（状态 + 日期 + 楼层 + 学生名）的 WHERE 条件组合很多，为每个组合建索引不现实。课堂项目数据量小（< 200 条预约记录），SQL Server 选择全表扫描或单索引扫描的性能差异可以忽略。如果未来数据量超过万级，建议在 Reservation 表上建 `INCLUDE` 索引来覆盖高频筛选组合。

### D.5 【建议】在 §2.2 "体验账号"的处理方式中指定具体方案

**位置**：§2.2 表格第一行

// 使用 UPDLOCK + HOLDLOCK 防止并发重复预约

    .FromSqlRaw(@"
