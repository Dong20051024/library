# PRD Lite —— 图书馆座位预约系统

> **文档定位**：本文档是 PRD 的精简版本，介于"需求确认"和"技术设计"之间。目标是让开发者（也就是你）拿到后可以直接开始编码，不需要反复回来问需求。
>
> 前置输入：docs/01-项目立项单.md、docs/02-需求分析与MVP确认.md

---

## 1. 项目概述

开发一个轻量化的图书馆座位预约 Web 系统。学生可以在线查看座位状态、预约座位、取消预约；图书馆管理员可以通过后台管理座位和预约数据，确认签到和释放座位。

系统基于 ASP.NET Core MVC (.NET 8) + Razor 技术栈构建，使用 SQL Server LocalDB 作为数据库，EF Core 做 ORM，Bootstrap 做前端样式。不涉及支付、扫码、消息推送、真实用户注册等复杂功能，不部署到公网服务器。

---

## 2. 项目目标

1. **核心链路跑通**：学生能完整走完"查看座位 → 预约座位 → 签到入座 → 释放座位"的全流程。
2. **后台可管可控**：管理员能登录后台，管理座位信息，查看和处理预约记录。
3. **单人可交付**：项目代码完整提交到 GitHub/Gitee 仓库，别人拉下来能直接运行。
4. **课堂时间可控**：6 天周期内交付 MVP（P0 + P1 功能），不超期不超范围。

---

## 3. 目标用户

| 角色 | 身份 | 系统使用方式 | 核心诉求 |
|------|------|-------------|----------|
| 学生（读者） | 在校大学生 | 浏览器访问用户端页面，通过体验账号切换模拟不同人 | 查座位、约座位、取消预约 |
| 管理员 | 图书馆工作人员 | 浏览器访问管理端，使用固定账号 + Session 登录 | 管理座位、处理预约、查看统计 |

## 4. 使用场景

| 场景 | 角色 | 频率 | 说明 |
|------|------|------|------|
| 预约座位 | 学生 | 高频 | 查看空闲座位 → 选择时段 → 提交预约 |
| 取消预约 | 学生 | 中频 | 在"我的预约"中对未签到记录执行取消 |
| 签到确认 | 管理员 | 中频 | 学生到馆后，管理员在后台将"待签到"改为"使用中" |
| 释放座位 | 管理员 | 中频 | 学生离开后，管理员将"使用中"改为"空闲" |
| 座位管理 | 管理员 | 低频 | 新增/编辑/禁用座位 |
| 数据查看 | 管理员 | 低频 | 查看预约趋势和座位利用率（P2，视时间） |

## 5. 用户端核心功能

> 每个功能附带最小验收标准。验收标准是"能演示"级别的，不是"通过全部测试用例"级别。

### F-U1：首页

| 项目 | 内容 |
|------|------|
| 描述 | 系统入口页面，展示系统名称、欢迎语和导航入口 |
| 最小验收标准 | 1. 页面打开后能看到"图书馆座位预约系统"标题<br>2. 有链接可以进入"座位列表"和"我的预约"<br>3. 能切换到不同的体验账号 |
| 路由 | `/` 或 `/Home/Index` |

### F-U2：座位列表

| 项目 | 内容 |
|------|------|
| 描述 | 展示所有座位，用颜色区分座位状态，支持筛选 |
| 最小验收标准 | 1. 显示所有座位，每个座位能看到座位号和状态<br>2. 空闲（绿色）、已预约（黄色）、使用中（红色）三种颜色区分<br>3. 支持按楼层筛选（如：1楼、2楼、3楼）<br>4. 点击空闲座位能跳转到预约页面<br>5. 已预约和使用中的座位不可点击预约 |
| 路由 | `/Seat/List` |

### F-U3：座位详情

| 项目 | 内容 |
|------|------|
| 描述 | 查看单个座位的详细信息，包括位置、设施标签 |
| 最小验收标准 | 1. 显示座位号、楼层、区域位置<br>2. 显示设施标签（如：有插座、靠窗、电脑位）<br>3. 显示座位当前状态<br>4. 如果座位空闲，显示"预约"按钮 |
| 路由 | `/Seat/Detail/{id}` |

### F-U4：预约提交

| 项目 | 内容 |
|------|------|
| 描述 | 选择预约时段并提交预约请求 |
| 最小验收标准 | 1. 页面显示所选座位的编号和信息<br>2. 提供日期选择和时段选择（或起止时间选择）<br>3. 提交成功后跳转到"我的预约"页面<br>4. 如果该座位在该时段已被预约，给出提示并阻止重复提交<br>5. 提交后对应座位在座位列表中变为"已预约"状态 |
| 路由 | `/Reservation/Create?seatId={id}` |

### F-U5：我的预约

| 项目 | 内容 |
|------|------|
| 描述 | 当前体验账号的所有预约记录列表 |
| 最小验收标准 | 1. 展示当前账号的所有预约记录<br>2. 每条记录显示：座位号、预约时段、状态<br>3. 不同状态用不同颜色或标签区分<br>4. "待签到"状态的预约可以点击取消<br>5. 如果没有任何预约，显示"暂无预约记录"的提示 |
| 路由 | `/Reservation/Mine` |

### F-U6：取消预约

| 项目 | 内容 |
|------|------|
| 描述 | 对"待签到"状态的预约执行取消操作 |
| 最小验收标准 | 1. 仅"待签到"状态的预约可以取消<br>2. 取消后预约状态变为"已取消"<br>3. 取消后对应座位恢复为"空闲"状态<br>4. 取消操作有二次确认（弹窗或提示） |
| 路由 | 在"我的预约"页面内执行 |

### F-U7：体验账号切换

| 项目 | 内容 |
|------|------|
| 描述 | 通过下拉选择或点击切换不同的体验账号，模拟不同学生登录 |
| 最小验收标准 | 1. 页面顶部或导航栏显示当前体验账号的名称<br>2. 点击后弹出可选账号列表（3-5 个）<br>3. 切换后"我的预约"中的数据变为新账号的数据<br>4. 切换后当前页面刷新或重定向到首页 |
| 路由 | 全局导航栏组件 |

## 6. 管理端核心功能

### F-A1：管理员登录

| 项目 | 内容 |
|------|------|
| 描述 | 使用固定账号 + 密码，通过 Session/Cookie 登录管理后台 |
| 最小验收标准 | 1. 访问管理端页面自动跳转到登录页<br>2. 输入正确的账号密码后登录成功，跳转到预约管理页<br>3. 输入错误的账号密码显示错误提示<br>4. 登录后页面右上角显示管理员名称<br>5. 有关闭浏览器后再次打开需要重新登录的机制（Session） |

### F-A2：预约管理

| 项目 | 内容 |
|------|------|
| 描述 | 查看所有预约记录，支持按状态筛选 |
| 最小验收标准 | 1. 展示所有预约记录（按创建时间倒序显示）<br>2. 每条记录显示：学生名、座位号、时段、状态<br>3. 支持按状态筛选（全部/待签到/使用中/已取消/已完成）<br>4. 对"待签到"的记录有"确认签到"按钮<br>5. 对"使用中"的记录有"释放座位"按钮 |

| 项目 | 内容 |
|------|------|
| 描述 | 管理员手动将预约从"待签到"改为"使用中" |
| 最小验收标准 | 1. 在预约管理页，点击"确认签到"按钮<br>2. 预约状态从"待签到"变为"使用中"<br>3. 对应座位状态从"已预约"变为"使用中"<br>4. 操作有成功提示 |

### F-A4：释放座位

| 项目 | 内容 |
|------|------|
| 描述 | 管理员手动将预约从"使用中"改为"已完成"，座位恢复空闲 |
| 最小验收标准 | 1. 在预约管理页，点击"释放座位"按钮<br>2. 预约状态从"使用中"变为"已完成"<br>3. 对应座位状态从"使用中"变为"空闲"<br>4. 操作有成功提示 |

| 项目 | 内容 |
|------|------|
| 描述 | 管理员对座位进行增删改操作 |

| 路由 | `/Admin/Seat` |

### F-A6：数据统计（P2 - 视时间决定）

| 项目 | 内容 |
|------|------|
| 描述 | 展示预约趋势图和座位利用率统计数据 |
| 最小验收标准 | 1. 显示近一周每天的预约数量趋势（折线图或柱状图）<br>2. 显示整体座位利用率百分比<br>3. 数据显示来自真实预约记录 |

## 7. 页面清单

> 共 **9 页**，其中 8 页为 MVP 必做（P0+P1），1 页为可选（P2）。路由与页面一一对应。

| # | 页面 | 路由 | 所属端 | 优先级 | 备注 |
|---|------|------|--------|--------|------|
| 1 | 首页 | `/` 或 `/Home/Index` | 用户端 | P1 | 系统入口，含体验账号切换组件 |
| 2 | 座位列表 | `/Seat/List` | 用户端 | P0 | 核心入口页，含筛选功能 |
| 3 | 座位详情 | `/Seat/Detail/{id}` | 用户端 | P1 | 座位信息 + "预约"按钮入口 |
| 4 | 预约提交 | `/Reservation/Create?seatId={id}` | 用户端 | P0 | 选时段 + 确认提交 |
| 5 | 我的预约 | `/Reservation/Mine` | 用户端 | P1 | 预约记录列表，含取消操作 |
| 6 | 管理员登录 | `/Admin/Login` | 管理端 | P0 | 固定账号 + Session 登录 |
| 7 | 预约管理 | `/Admin/Reservation` | 管理端 | P0 | 含签到确认和释放座位操作 |
| 8 | 座位管理 | `/Admin/Seat` | 管理端 | P1 | 座位的增删改 |
| 9 | 数据统计 | `/Admin/Statistics` | 管理端 | **P2** | **可选**，时间充裕再加 |

**页面结构说明**：

- 用户端 5 个页面共用同一套 Bootstrap 导航布局（_Layout.cshtml）
- 管理端 3 个页面共用一套独立的后台布局，路由集中在 `/Admin/*` 前缀下
- 体验账号切换组件放在用户端布局的导航栏中，所有用户端页面均可访问
- 总共 9 页 = 5 用户端 + 3 管理端 MVP + 1 统计页（可选）

## 8. 核心业务流程

### 8.1 核心预约链路

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐

                                                              ┌──────────────┐
                                                              │ 取消预约      │
                                                              │ (待签到→已取消)│
```

```
┌──────────┐    ┌──────────┐    ┌──────────┐
│ 管理员登录 │ →  │ 预约管理  │ →  │ 签到确认   │ → 释放座位

```

### 8.3 完整业务时序

```
学生                     系统                    管理员
  │                       │                       │
  ├── 切换体验账号 ──────→ │                       │
  ├── 查看座位列表 ──────→ │                       │
  │                       │ 返回座位及状态          │
  ├── 选择空闲座位 ──────→ │                       │
  ├── 提交预约 ──────────→ │                       │
  │                       │ 创建预约(待签到)        │
  │                       │ 锁定座位               │
  │  ← 预约成功 ──────────┤                       │
  │                       │                       │
  │ (到馆)                 │                       │
  │                       │  ← 管理员确认签到 ──── │
  │                       │ 状态改为"使用中"        │
  │                       │                       │
  │ (离馆)                 │                       │
  │                       │  ← 管理员释放座位 ──── │
  │                       │ 状态改为"已完成"        │
  │                       │ 座位恢复空闲            │
```

## 11. 核心数据对象

> 本期只需要 3 张核心数据表。以下为业务字段设计，具体数据库类型和约束在技术设计阶段确定。

### 11.1 Seat（座位）

| 字段 | 类型 | 说明 |
|------|------|------|
| Id | int | 主键，自增 |
| Floor | string | 楼层（如："3楼"） |
| Area | string | 区域（如："A区"、"靠窗区"） |
| SeatNumber | string | 座位编号（如："A-01"） |
| Status | enum | 座位状态：空闲 / 已预约 / 使用中 |
| Facilities | string | 设施标签（如："有插座,靠窗,电脑位"，逗号分隔） |
| IsActive | bool | 是否启用（禁用后不可预约） |

### 11.2 Reservation（预约）

| 字段 | 类型 | 说明 |
|------|------|------|
| Id | int | 主键，自增 |
| SeatId | int | 外键 → Seat |
| StudentName | string | 预约学生姓名（体验账号的用户名） |
| StartTime | DateTime | 预约开始时间 |
| EndTime | DateTime | 预约结束时间 |
| Status | enum | 预约状态：待签到 / 使用中 / 已完成 / 已取消 |
| CreatedAt | DateTime | 创建时间 |

### 11.3 Admin（管理员）

| 字段 | 类型 | 说明 |
|------|------|------|
| Id | int | 主键，自增 |
| Username | string | 登录账号名 |
| Password | string | 密码（本期明文存储，不做加密） |
| DisplayName | string | 显示名称 |

> 密码本期不加密，是因为课堂项目阶段不做安全攻防演练，且数据只在本地 LocalDB 中。如果课程有安全要求，管理员密码可以用简单哈希。

## 12. 状态流转说明

### 12.1 座位状态机

```
                 预约提交                管理员签到
   [空闲] ─────────────────→ [已预约] ────────────→ [使用中]
     ↑                                                 │
     │                  取消预约                        │ 管理员释放
     └──────────────── [已取消] ←──── (取消操作) ──────┘
                                                        │
                                                        ▼
                                                  [已完成] ←── 释放
```

### 12.2 预约状态机

```
  提交预约                管理员签到                管理员释放
[待签到] ─────────────→ [使用中] ──────────────→ [已完成]
    │
    │ 用户取消
    ▼
 [已取消]
```

### 12.3 状态流转规则

| 当前状态 | 允许的操作 | 目标状态 | 执行角色 |
|----------|-----------|----------|----------|
| 空闲 | 提交预约 | 已预约 | 学生 |
| 已预约 | 确认签到 | 使用中 | 管理员 |
| 使用中 | 释放座位 | 已完成 / 空闲 | 管理员 |
| 待签到 | 取消预约 | 已取消 | 学生 |
| 待签到 | 确认签到 | 使用中 | 管理员 |

### 12.4 禁止的流转

| 操作 | 为什么禁止 |
|------|-----------|
| 学生取消"使用中"的预约 | 已经入座了不能线上取消，需管理员处理 |
| 重复预约同一时段同一座位 | 时段内座位已被锁定 |
| 删除有预约记录的座位 | 会导致数据不一致 |
| 已取消/已完成的预约再次操作 | 生命周期已结束 |

## 14. 暂不开发内容

> 以下功能已明确排除在本期之外。开发过程中如果产生"要不要顺手加上"的念头，请回到这里确认。

### 从 MVP 中删减的功能

| 功能 | 删减原因 |
|------|----------|
| 学生自助签到（扫码/输入码） | 由管理员后台代劳，简化开发 |
| 座位地图可视化 | 列表模式 + 颜色区分足矣，地图开发成本高 |
| 收藏常用座位 | 非核心需求 |
| 开放时间/公告管理 | 偏离核心预约流程 |

| 操作日志/审计 | 课堂项目不需要 |
| 预约历史归档 | 有"我的预约"即可查看近期记录 |

### 立项阶段已排除的功能

| 功能 | 排除理由 |
|------|----------|
| 支付/押金/罚款 | 复杂度高，非核心 |
| 扫码签到/扫码抢座 | 本期走管理员手动确认 |

| 黑名单/信用分 | 运营功能，初始不需要 |
| 第三方 OAuth 登录 | 用体验账号切换替代 |

## 15. 风险与假设

### 风险

| 风险 | 可能性 | 影响 | 应对措施 |
|------|--------|------|----------|
| 页面数量失控，越加越多 | 中 | 高 | 页面清单锁定 9 页，新增需求一律列入"暂不开发" |
| EF Core 迁移失败，数据库表和模型对不上 | 中 | 高 | 严格按 Add-Migration → Update-Database 流程，不手动改库 |
| 预约时段冲突逻辑有 bug | 中 | 中 | 后端做时段重叠校验，不依赖前端判断 |
| 状态流转混乱（学生取消已签到的预约） | 低 | 中 | 按状态机严格实现，前端后端双重校验 |
| LocalDB 本地安装有问题 | 低 | 高 | 提前确认环境；实在不行换成 SQLite |
| AI 生成的代码改出问题 | 中 | 中 | 每次改代码前 git commit，出问题直接回滚 |

### 假设

- 开发环境为 Windows 11 + Visual Studio 2022 + .NET 8 SDK
- SQL Server LocalDB 可用（随 Visual Studio 安装附带）
- 演示时使用浏览器直接操作，不需要部署到服务器
- 并发场景不复杂，两个学生同时约同一个座位时后提交的报错提示即可
- 种子数据预置足够演示使用（3-5 个体验账号、10+ 个座位、若干预约记录）
